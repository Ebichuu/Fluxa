# 媒体控制中心 PT 主链收口设计

状态：待用户复核

日期：2026-07-17

范围：当前代码阶段，不执行 fnOS 真实写动作

## 1. 目标

当前版本先把媒体控制中心的 PT 主线路收口，不继续开发 Telegram 频道驱动的网盘订阅、资源搜索或自动兜底。

固定主线路：

```text
媒体控制中心订阅
  → Torra 搜索与下载编排
  → qBittorrent 下载
  → Torra 秒传到 115
  → Symedia 整理、STRM 与归档
  → Emby 索引与影院大厅
```

媒体控制中心是唯一用户入口，但不重复实现外部软件已经负责的业务。它负责订阅台账、直接推送、安全控制、状态聚合、任务关联和异常解释。

## 2. 业务所有权

| 能力 | 唯一负责人 | 当前中控职责 |
| --- | --- | --- |
| 内容发现、订阅、日历、分类和资源规则 | NasEmby Python 源码 | 提供统一 React 页面和安全 API |
| PT 搜索、站点规则和下载编排 | Torra | 推送订阅、读取状态、关联任务 |
| 下载 | qBittorrent | 读取进度，提供已存在的暂停/恢复安全动作 |
| qB 下载后的 115 秒传 | Torra | 读取可获得的秒传证据，不自行启动第二套上传器 |
| 115 内容整理、STRM 和归档 | Symedia | 只读取处理记录和入库证据，不推送订阅 |
| 媒体库索引 | Emby | 读取媒体证据和影院大厅数据，保留受保护刷新 |
| Torra 无结果后的其他 PT 站点补齐 | MoviePilot（未来） | 当前只保留状态和源码，不自动参与主线 |

NasEmby 的 `app/legacy/ptto115.py` 继续保留，但当前生产运行不启动。Torra 已经负责 qB 到 115 的秒传；同时启动两套上传器会造成重复秒传、文件争抢和删除时序风险。

| Torra 推送预览与在线查重 | `app/subscription_compat_runtime.py` 的现有预览流程 |

## 4. 订阅与直接推送

用户仍只在媒体控制中心创建和管理订阅。保存后写入唯一 NasEmby 台账：

- `db/discover_subscription_items.json`
- `db/discover_subscriptions.json`

订阅详情只增加 **检查并推送到 Torra**，不提供 Symedia 或 MoviePilot 订阅推送按钮。

操作流程：

1. 浏览器请求只读预览。
2. 服务端从唯一台账重新读取订阅，不接受浏览器提交的任意媒体对象。
3. 预览显示标题、TMDB ID、媒体类型、季号、统一分类、目标服务和已知阻塞原因。
4. 用户确认后提交提供者、确认值和幂等键。
5. 服务端重新读取订阅并复查目标服务状态。
6. 调用原 `push_subscription_to_provider(item, "torra")`。
7. Torra 原函数执行已有订阅查重；结果写入活动日志。
8. 前端刷新订阅和任务链，不把“请求已接受”伪装成“已经下载”。

Torra 继续要求有效分类、下载器 ID、保存路径和在线查重。Symedia 只承担 115 后的整理与入库，MoviePilot 自动补齐留待未来单独设计。

## 5. PT 任务链

任务中心固定展示四个用户可理解的步骤：

```text
订阅 → Torra / qB → 进入 115 → Symedia / Emby
```

证据规则：

- **订阅**：唯一 NasEmby 台账存在稳定媒体身份。
- **Torra / qB**：按 TMDB ID、媒体类型、季号和下载文件名关联。
- **进入 115**：优先读取 Torra 可提供的秒传结果或 Symedia 已接管记录。
- **Symedia / Emby**：使用 Symedia 转存历史和 Emby ProviderId。

若当前 Torra API 没有提供单条秒传结果，只能把 qB 已完成且随后出现 Symedia 记录标为“相邻证据”。只有明确成功记录才能显示“已验证”；下载完成本身不能证明已经进入 115。

现有响应中的 `cloud115` 步骤键为兼容客户端继续保留，但页面名称固定为“进入 115”，不得再解释为 Telegram 网盘获取通道。

## 6. Telegram 网盘能力延期

下列能力移入未来版本：

- Telegram 登录、频道资源订阅和监控。
- HDHive / pansou、影巢搜索与解锁。
- 115 分享链接转存。
- PT 失败后的网盘候选和自动兜底。
- 115 清理和助力。

处理方式：

- React 当前不展示相关设置、候选按钮、订阅级开关或任务状态。
- Python 源码、原接口、v2 安全适配层、测试和能力矩阵继续保留。
- 所有相关环境闸门保持关闭。
- 不删除旧调用关系，也不启动后台自动任务。
- 订阅配置中已有网盘字段原样保留，页面保存其他字段时不得覆盖或丢失。

## 7. MoviePilot 后续 PT 补齐

MoviePilot 属于 PT 补齐通道，不属于 Telegram 网盘通道。

未来流程：

```text
Torra 明确无结果或达到等待条件
  → 再次确认没有 Torra / qB 活动任务
  → MoviePilot 搜索其他 PT 站点
  → 继续进入同一 qB → 115 → Symedia → Emby 主线
```

当前只保留 `moviepilot_status()`、`moviepilot_subscribe()` 和配置字段，不在当前界面提供推送入口。自动补齐必须在后续单独设计，届时以 [MoviePilot 官方文档](https://wiki.movie-pilot.org/) 和现有源码为准，不能让 Torra 与 MoviePilot 并行重复下载。

## 8. 页面与导航

影院大厅继续沿用 Mineradio 的核心视觉、封面和交互，但顶部导航的按钮结构不再冻结。

主导航调整为：

```text
总览 / 影院大厅 / 发现 / 订阅 / 任务中心 / 日历
```

- “订阅”成为直接入口，打开我的订阅；订阅设置仍从订阅页内部进入。
- “控制室”不再占用重复的主按钮；右侧服务健康按钮继续进入控制室。
- 搜索媒体保留为右侧快捷动作。
- 设置保留齿轮入口。
- 视觉语言、尺寸和响应式基础继续沿用现有导航，不进行无关换肤。

内容发现与订阅共享 NasEmby 数据和组件，但路由状态分开，用户返回时能够保留各自筛选条件。

## 9. 影院大厅媒体队列

现有抽屉实际展示“媒体库”和“当前媒体库内容”，不是 Emby 播放队列。优化必须先修正语义：

- “当前队列”改为“本库内容”。
- 保留媒体库切换、项目选择、当前焦点和常开状态。
- 当前项目变化时确保列表能定位到选中项。
- 移动端增加明确的打开与关闭操作，不依赖鼠标悬停。
- 空状态区分“媒体库未返回”和“本库没有可展示内容”。
- 不把 PT 下载任务、订阅任务或 Emby 播放会话塞进该抽屉；这些仍属于任务中心。

本轮不修改 Mineradio 内部 Three.js、GSAP、封面视觉和媒体选择桥接协议。

## 10. 活动日志与系统指标

### 活动日志

任务中心增加最近活动时间线，读取原 `read_activities()`：

- 默认最近 100 条。
- 支持按订阅、推送、qB、系统分类筛选。
- 不展示 Cookie、Token、完整 qB hash、完整保存路径或上游异常正文。
- 本轮不提供清空日志按钮。

### 系统指标

总览增加 CPU、内存、磁盘和网络摘要，调用原 `dashboard_system_metrics()`，再经过字段白名单映射。

原函数会进行 CPU 和网络采样并读取 Emby 库，因此：

- 服务端缓存至少 30 秒。
- 前端仅在总览可见时读取，自动刷新间隔不短于 60 秒。
- 指标失败只影响该卡片，不使总览或其他业务接口失败。
- 不向浏览器返回内部文件路径或 Emby 凭据。

## 11. API 边界

新增或转为浏览器客户端的接口必须使用细分契约，不能开放原核心总开关：

- Torra 推送预览：读取订阅与 Torra 状态，无外部写入。
- Torra 推送动作：目标固定为 `torra`，要求确认、幂等键、服务端复查和独立写闸门。
- 系统指标：只读、缓存、字段白名单。
- 活动日志：继续使用现有只读接口并加入前端契约。

浏览器不能提交完整 NasEmby 条目、提供者凭据、115 Cookie 或任意外部 URL。错误响应使用固定错误码、用户可理解的说明和请求 ID。

## 12. 性能与运行模型

- 不增加新的常驻调度线程。
- 不启动 NasEmby `ptto115.py`。
- Torra 直接推送只在用户操作时执行。
- 日志读取限制条数。
- 系统指标使用缓存和低频轮询。
- Gunicorn 继续保持一个 worker、四个线程。
- 当前 JSON 台账仍是单实例模型，不在本轮引入 SQLite 或多副本。

## 13. 安全边界

- 生产默认关闭所有真实写闸门。
- 危险方法继续校验同源。
- Torra 推送在服务端按订阅 ID 重新读取目标，不能使用浏览器伪造载荷。
- 相同幂等键只执行一次；同订阅短时间重复推送进入冷却。
- 只记录脱敏活动，不记录完整凭据、分享链接或外部异常正文。
- 保留核心路由继续默认返回 `503 PRESERVED_CORE_API_DISABLED`。

## 14. 测试与验收

自动测试只使用临时台账和模拟外部服务：

1. Torra 预览不产生写调用，界面没有 Symedia 或 MoviePilot 推送入口。
2. Torra 推送的目标固定、确认、幂等、冷却和默认闸门逐项拒绝错误请求。
3. 执行时从唯一台账重建条目，不信任浏览器载荷。
4. Torra 原函数只被调用一次，结果正确脱敏并写入活动日志。
5. 任务中心继续区分 115 已验证、相邻证据和缺失。
6. React 不再引用 Telegram/HDHive/影巢/网盘兜底客户端动作。
7. 网盘底层源码、路由和原测试仍然存在。
8. 导航新增订阅入口、健康按钮进入控制室，移动端无横向溢出。
9. 媒体抽屉键盘、触摸、固定和当前项目定位有效。
10. 系统指标缓存生效，局部失败不会拖垮总览。
11. Python 测试、TypeScript、Vite 构建、npm 审计和安全扫描通过。

代码阶段不连接真实 Torra、qB、115、Symedia、MoviePilot 或 Emby 写接口。

## 15. 完成标准

本轮完成后应满足：

- 用户可以从媒体控制中心发现内容、创建订阅并直接进入我的订阅。
- 单条订阅可以安全预览并人工推送到 Torra；Symedia 不接收重复订阅推送。
- PT 主链在任务中心按真实证据展示到 Emby。
- Torra 独占 PT/qB/115 秒传，Symedia 独占整理入库，中控没有重复业务实现。
- Telegram 网盘相关界面不再干扰当前 PT 使用，但源码和后续恢复路径完整保留。
- MoviePilot 后续 PT 补齐方向有明确记录，当前不会自动并行执行。
- 导航和媒体抽屉更符合实际用途，不破坏 Mineradio 影院大厅主体视觉。

## 16. 实机阶段（本轮不执行）

代码验收完成后，未来 fnOS 实机按以下顺序验证：

1. 只读部署并核对 Torra、qB、Symedia、Emby 状态。
2. 核对 Torra 下载器 ID、八分类保存路径和 115 秒传目标。
3. 创建一条电影测试订阅并预览 Torra 载荷。
4. 单独批准一次 Torra 真实推送。
5. 观察 qB 下载、Torra 秒传、Symedia 接管和 Emby 索引。
6. 电影闭环成立后再验证一条剧集季。
7. 主链稳定后才开启订阅调度；MoviePilot 自动补齐和 Telegram 网盘功能继续保持关闭。
