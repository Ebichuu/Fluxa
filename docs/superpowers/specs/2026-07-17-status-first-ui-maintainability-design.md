# 状态首屏与前端可维护性设计

日期：2026-07-17

## 1. 目标

媒体控制中心的工作页首先回答“系统现在怎么样、我是否需要处理”，不再用大号产品宣言占据首屏。同时收口当前 5431 行单文件 CSS、文案层级、字体混排、色彩对比度和工作页键盘导航问题。

本设计不改变订阅台账、API 契约、PT 主线或外部写入闸门。影院大厅整体冻结：React 组件、Mineradio / Three.js、媒体队列、状态条、交互和大厅专属 CSS 均不修改。浅色模式不在本轮实现，继续保留为低优先级路线。

## 2. 状态首屏

### 2.1 共享结构

所有工作页使用共享 `PageStatusHeader`，结构固定为：

```text
页面身份                当前状态                         操作
总览                    4 项需要检查                    查看任务中心
PT 主链                 订阅 12 · 下载 3 · 今日入库 5
```

- `<h1>` 只放页面名，例如“总览”“任务中心”，字号 18–22px。
- 当前状态是首屏视觉重点，字号 30–38px，但不是宣传句。
- 补充行只放实时数字、更新时间或直接原因。
- 右侧最多放一个主操作和一个刷新操作。
- 移动端按“页面名 → 当前状态 → 操作”纵向排列。
- 删除工作页的 `.ops-eyebrow` 和 `.ops-deck`；卡片内部的小标不受影响，但必须描述数据含义。

共享组件只负责结构和语义，不获取数据：

```ts
interface PageStatusHeaderProps {
  title: string;
  context?: string;
  status: ReactNode;
  detail?: ReactNode;
  tone?: 'ok' | 'warn' | 'neutral';
  actions?: ReactNode;
}
```

每个页面继续拥有自己的请求和状态转换，避免共享组件变成新的业务中枢。

### 2.2 页面状态

| 页面 | 主状态 | 补充信息 | 首屏操作 |
| --- | --- | --- | --- |
| 总览 | `PT 主链正常` 或 `4 项需要检查` | 订阅数、活跃下载、今日入库 | 查看任务中心 |
| 任务中心 | `0 个进行中 · 0 个卡住` | 最近更新时间；读取中和失败要明确显示 | 刷新 |
| 控制室 | `0 / 4 核心服务在线` | 需检查数量或全部正常 | 刷新全部服务 |
| 日历 | `本月 0 个待播 · 0 个逾期` | 当前年月、已入库数量 | 无；月份操作留在月历 |
| 发现 | `全球日播 · 0 条结果` | 当前搜索词或筛选数量 | 聚焦搜索 |
| 我的订阅 | `0 条订阅 · 0 条未完成` | PT 优先、最近执行时间 | 订阅设置 |
| 订阅设置 | 当前通道，例如 `PT / Torra 主通道` | 自动订阅状态和执行时间；数据未就绪时只显示当前通道 | 返回我的订阅 |
| 系统设置 | `1 / 10 项服务已配置` | 访问保护状态 | 无 |

加载时使用“正在读取任务状态”等真实状态；失败时直接显示“任务状态读取失败”，不回退成口号或模糊说明。

## 3. CSS 模块化

### 3.1 文件结构

`src/main.tsx` 改为只导入 `src/styles/index.css`。`index.css` 按以下顺序导入：

```text
styles/
  index.css
  foundation.css       令牌、字体、reset、基础元素、无障碍工具
  shell.css            顶部导航、应用外壳、登录外层
  global.css           原位置保留的影院大厅冻结样式，不做规则修改
  workbench.css        工作页背景、状态首屏、共享面板和按钮
  overview.css
  discover.css         发现、我的订阅、资源详情与 Torra 预检
  tasks.css
  calendar.css
  control-room.css
  settings.css         系统设置和订阅设置
```

响应式规则跟随所属模块，不单独放入一个巨型 `responsive.css`。`prefers-reduced-motion` 的工作页规则放在 `foundation.css` 或所属模块。影院大厅现有响应式与 reduced-motion 规则继续留在 `global.css`，内容和顺序不变。

### 3.2 迁移原则

1. 先标记影院大厅冻结选择器：`.media-hall*`、`.mineradio-*`、`.media-queue-*` 及其响应式规则不得编辑、重排或清理。
2. 只把非大厅样式按现有级联顺序机械抽离，不同时重命名全部类名。
3. `index.css` 在原级联位置导入保留的 `global.css`，Vite 生产构建仍输出合并压缩后的 CSS。
4. 拆分提交完成视觉等价验证后，只删除工作页旧分区中已被 `ops-*` 覆盖的重复规则；大厅规则不参与死代码清理。
5. 状态首屏与无障碍改造使用独立提交，出现回归时可以单独回退。
6. 每个工作页样式只能修改自己的根命名空间或共享组件；禁止新的无前缀通用选择器。

拆分的主要收益是定位、审查和控制级联，不把它描述成显著的运行时加载优化。

## 4. 文案层级检查

`UI_STANDARD.md` 增加提交前检查清单：

- `<h1>` 是否只是页面名，而非宣言、愿景或使用说明。
- 首屏最醒目的文字是否来自实时状态。
- 状态是否说明正常、等待、失败或需要处理。
- 一个标签是否只做标签工作，不同时承担说明和宣传。
- 工具名、API、路径和匹配依据是否放在次级层。
- 空状态是否说明下一步，而非只说“暂无数据”。
- 同一动作的按钮、确认框和结果提示是否使用同一个动词。

`.ops-eyebrow` 从工作页标准中移除；后续代码扫描不允许工作页重新引入。

## 5. 字体与中英文混排

不引入网络字体。基础令牌统一为：

```css
@font-face {
  font-family: "MCC Latin Mono";
  src:
    local("JetBrains Mono"),
    local("Cascadia Mono"),
    local("Consolas");
  unicode-range: U+0020-007E, U+00A0-024F;
}

:root {
  --font-ui: "Segoe UI", "Microsoft YaHei UI", "Noto Sans SC", sans-serif;
  --font-code: "MCC Latin Mono", var(--font-ui);
}
```

- 工作页正文、页面身份、状态和按钮使用 `--font-ui`；影院大厅、媒体队列和大厅状态条继续使用现有字体规则。
- `code`、`pre`、路径、环境变量、API 和 ID 使用 `--font-code`。
- 中文字符不落入 Latin Mono 的 `unicode-range`，自动回退到 `--font-ui`。
- 速度、数量等业务数字继续使用 UI 字体，并通过 `font-variant-numeric: tabular-nums` 对齐，不为数字滥用代码字体。
- 删除页面内重复的字体栈和 `.ops-eyebrow` 等宽声明。

## 6. 对比度

- 在 `.ops-page` 范围建立可验证的工作页文字颜色，不直接修改影院大厅继续使用的根级 `--muted` 与 `--faint`。
- 普通正文、状态说明、错误和路径按 WCAG AA 4.5:1 验证。
- 18px 以上大字最低 3:1，但状态首屏仍以 4.5:1 为目标。
- 低于标准的透明度只允许用于边框、光晕和不承载信息的装饰。
- 禁用状态可以降低强调，但必须同时通过形状、文案或 `disabled` 语义表达，不只依赖低对比度。

最终颜色由自动对比度脚本针对 `#010304`、普通面板和控制面板三种实际底色计算后确定；验收记录写入实施文档，不能凭肉眼判断。

## 7. 键盘导航

### 7.1 影院大厅

- 本轮不修改 `MediaHall.tsx`、`MineradioEmbed.tsx`、`MediaQueuePanel.tsx` 或 `vendor/mineradio-public/**`。
- 不修改大厅的方向键、滚轮、舞台焦点、媒体抽屉、状态条、iframe 通信或视觉设置。
- 不修改、重排或清理 `.media-hall*`、`.mineradio-*`、`.media-queue-*` 大厅专属样式。
- 大厅键盘导航和其他可访问性问题只保留记录，后续必须单独设计并再次取得用户确认。

### 7.2 工作页

- 导航、筛选标签、日历类型、订阅类型和季选择使用标准标签键盘模式。
- 所有可操作元素都有可见 `focus-visible`，颜色不能低于 3:1。
- 确认框打开后焦点进入对话框，`Escape` 关闭，关闭后焦点返回触发按钮。
- 纯图标按钮必须有中文可访问名称。

## 8. 数据流与错误处理

`PageStatusHeader` 不新增 API。页面把已有请求结果转换为状态模型后传入：

```text
现有 API / 页面 state
  → 页面内状态计算
  → PageStatusHeader 展示
```

系统设置需要显示配置数量时，由 `App` 将现有 `health` 结果传给 `SettingsPage`，不重复请求健康接口。订阅设置如果暂时拿不到调度状态，显示已有通道状态，不猜测“已启用”。

加载、失败、空数据和正常数据必须是四种独立状态。页面请求失败时保留刷新入口，不把失败显示成零。

## 9. 验证

### 自动检查

- TypeScript 类型检查与 Vite 生产构建。
- Python 既有 85 项回归测试。
- CSS 入口和模块依赖扫描，禁止模块间循环导入。
- 工作页源码扫描：禁止 `.ops-eyebrow`，每页只能有一个页面名 `<h1>`。
- 对比度脚本覆盖三种实际背景。
- `npm audit --omit=dev` 与 `git diff --check`。

### 浏览器验收

- 1440×900、1024×768、390×844 三种尺寸。
- 总览、任务、控制室、日历、发现、订阅和两类设置的状态首屏。
- 从顶部导航开始，只用 Tab、Shift+Tab、方向键、Enter、Space 和 Escape 完成工作页核心流程；影院大厅键盘行为不在本轮验收范围内。
- 中英混合路径、环境变量、API 和中文说明的字体回退正确。
- 浏览器控制台无新增错误；影院大厅截图、交互和计算样式与本轮开始前保持一致。

## 10. 实施顺序

1. 建立影院大厅冻结清单，再建立 CSS 模块入口，只机械抽离非大厅样式并完成视觉等价验证。
2. 建立字体与文字颜色令牌，完成对比度脚本。
3. 新增共享状态首屏，逐页替换宣言式 Hero。
4. 修复工作页标签、确认框和通用焦点样式；影院大厅保持现状。
5. 删除确认无引用的旧 CSS，更新 UI 标准与当前计划。
6. 完成全量自动检查和三尺寸浏览器验收。

浅色模式在以上工作稳定后另立设计，不与本轮混合。
