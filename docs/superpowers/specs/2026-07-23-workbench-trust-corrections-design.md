# 追更工作台可信状态与资源检索修正版设计

日期：2026-07-23
状态：已确认，待实施

## 背景

`v0.4.2` 已建立任务身份、三维状态、RSS 身份字段、海报失败占位和移动日历，但实机数据暴露了五类收口问题：

1. 正常保护条目显示为 `healthState=protected`，却仍可能得到 `executionState=action_required`。
2. `reasonText`、`userReasonText` 与 `technicalReasonText` 在部分阶段仍保存同一段原始路径和错误文本。
3. 首页问题项缺少季集身份，同一作品的多条异常看起来完全相同，且身份未关联可能遮住真正的执行失败。
4. RSS 搜索为了避免无关种子而同时强制 TMDB、类型、季号和年份，现有未识别条目因此几乎全部被过滤。
5. 追更卡片只实现了海报失败占位，没有为已有订阅补齐可可靠获得的海报地址。

当前约 650 条 RSS 记录仍全部处于未识别状态。这可能是来源没有公开外部 ID，也可能是历史数据尚未经过有效回填，因此需要独立的端到端验收来区分“链路不可用”和“来源确实没有身份”。

## 目标

- 保证对账状态、执行状态和健康状态三个维度互不覆盖且语义一致。
- 普通界面只展示用户可理解的原因，原始路径、Hash、URL 和外部错误仅进入技术详情。
- 首页问题项能准确定位作品、季、集和真正阻塞阶段。
- 在不恢复模糊全量搜索的前提下，让缺少 RSS 身份的可靠标题候选可以被找到。
- 使用明确 TMDB 身份补齐已有订阅海报，并避免标题模糊误配。
- 使用隔离固定样本验证 RSS 身份解析、持久化、回填和查询的完整链路。

## 非目标

- 不访问 PT 详情页，不新增下载、转存或 Torra 推送动作。
- 不把标题相似当作身份，不向多个候选中的任意一个强行绑定。
- 不向正式 RSS 种子库插入测试数据。
- 不因正常保护自动删除残留文件。
- 不删除或修改 Torra 远端订阅。

## 一、三维状态一致性

### 状态职责

- `identityState`：媒体身份是否已关联，取值保持 `linked`、`unidentified`、`conflict`。
- `executionState`：执行链当前进度，取值保持 `normal`、`waiting`、`protected`、`suspected_blocked`、`action_required`、`confirmed_failed`。
- `healthState`：用户是否需要介入，取值保持 `normal`、`waiting`、`protected`、`evidence_insufficient`、`action_required`。

### 裁决顺序

任务执行状态必须根据已分类阶段统一裁决，不能让旧的顶层 `state=blocked` 覆盖结构化阶段结果：

1. 存在经过验证的真实失败：`confirmed_failed`。
2. 存在真实失败但证据未验证：身份明确时为 `action_required`，身份不明时为 `suspected_blocked`。
3. 没有真实失败但存在保护性拒绝：`protected`。
4. 存在计划重试或正在处理：`waiting`。
5. 全部阶段成功：`normal`。
6. 其他情况保持等待或证据不足，不根据原始 `blocked` 单独判红。

`identityState` 始终独立计算，不因保护或失败被改写。当前四条已有明确媒体身份的正常保护记录验收结果必须为：

```text
identityState: linked
executionState: protected
healthState: protected
```

正常保护不得进入异常、确认失败或可重试统计。

## 二、用户原因与技术原因

### 字段规则

- `technicalReasonText`：保存来源返回的完整原始内容，可包含路径、Hash、内部 ID 和原始错误。
- `userReasonText`：根据 `reasonCode`、阶段和来源生成稳定中文说明，不直接复制原始文本。
- `reasonText`：为兼容旧客户端，固定等于 `userReasonText`。

生成用户原因时先使用结构化规则，再使用文本兼容层。不能依赖少量路径前缀正则来决定是否需要脱敏。

示例：

```text
reasonCode: SYMEDIA_LIBRARY_FAILED
userReasonText: Symedia 未查询到对应媒体信息
technicalReasonText: /完整/文件/路径.mkv 与原始 Symedia 错误
```

普通任务卡片、首页、追更页和阶段摘要只能读取 `userReasonText` 或兼容字段 `reasonText`。`technicalReasonText` 只在明确展开“技术详情”时显示。

### 安全要求

普通响应文案不得泄露：

- Windows、Linux、UNC 和云盘绝对路径；
- qB Hash、远端文件 ID 和内部任务 ID；
- 带鉴权参数的 URL；
- RSS passkey、Token、Cookie 和外部鉴权字段。

## 三、首页问题项定位

首页问题项的季集身份按以下优先级提取：

1. 真正的问题阶段或制品证据；
2. 标准 `targetKey`；
3. 任务顶层 `seasonNumber`、`episodeNumber`。

响应增加兼容性的可选字段：

- `displayTitle`，例如 `云月大陆 S01E05`；
- `seasonNumber`；
- `episodeNumber`；
- `secondaryReasonText`，用于显示身份未关联、证据时间等次要说明。

首页主标题和主原因优先反映执行事实：

```text
云月大陆 S01E05 入库识别失败
Symedia 未查询到对应媒体信息
次要说明：任务尚未关联到可靠媒体身份
```

身份状态不能覆盖已经存在的真实执行失败。首页点击仍必须携带 `chainId` 和 `targetKey` 精确进入对应任务时间线。

## 四、RSS 订阅资源分层搜索

### 第一层：明确身份

- RSS 条目的 TMDB ID 与目标一致：精确命中，`matchMethod=tmdb_exact`、`matchConfidence=strong`。
- RSS 条目存在不同 TMDB ID：直接排除。
- 明确身份命中仍需排除已知冲突的媒体类型或季号。

### 第二层：未识别条目的保守候选

只有 RSS 条目没有 TMDB/IMDb 身份时才进入标题回退：

#### 电视剧

- 标题必须命中规范标题或详情提供的明确别名；
- 媒体类型必须为 `tv`，或来源尚未识别类型但没有电影证据；
- 已识别季号必须等于目标季号；
- 季号为空时仅允许作为“季号待确认”候选，标题中存在冲突季号则排除；
- 不再要求种子标题包含电视剧首播年份。

#### 电影

- 标题必须命中规范标题或明确别名；
- 必须匹配年份，除非已有明确 TMDB 身份；
- 年份缺失或存在多个可靠候选时不返回为可靠命中。

回退结果标记为 `fallback`，只用于人工查看。它不能直接触发自动下载、转存或反向身份写入；只有身份回填流程确认唯一可靠目标后，才能升级为已识别记录。

### 查询安全

- 非空但无法生成搜索 token 的查询仍返回 0 条。
- 不恢复无条件最新种子列表。
- 服务端按 RSS 条目 ID 去重，前端最多提交四个明确别名。
- 返回 `matchMethod`、`matchConfidence` 和季号是否待确认，供界面解释为何出现该结果。

## 五、已有订阅海报补齐

追更工作台读取到 `posterUrl` 为空时，仅允许使用明确 TMDB ID 补齐：

1. 优先读取本地发现/TMDB 缓存；
2. 缓存无结果且 TMDB 已配置时，按明确 ID 获取元数据；
3. 本地订阅成功补齐后写回 SQLite，避免每次页面加载重复请求；
4. 仅 Torra 的只读条目可以在响应中显示按明确 ID 得到的海报，建立本地镜像后再持久化；
5. 没有明确身份、获取失败或地址失效时保留首字占位。

不得仅按模糊标题为订阅选择海报。所有外部图片继续通过允许列表代理加载，失败后只回退一次，不形成重试循环。

## 六、RSS 身份端到端验收

使用临时 SQLite 数据库执行三条固定样本，不接触正式 RSS 数据：

1. RSS 自定义字段直接包含 TMDB ID；
2. RSS 简介或公开链接包含 IMDb ID；
3. 没有外部 ID，但标题、类型、年份/季号只对应一个已有追更。

每条样本必须完整通过：

```text
RSS XML 解析
→ SQLite 写入
→ 未识别记录回填
→ 身份状态更新
→ API 按身份和范围查询
→ 前端可读取身份来源与置信度
```

另增加冲突样本，验证多个可靠追更候选时结果为 `conflict`，不得随机绑定。

正式数据的界面需要区分：

- 身份回填尚未运行；
- 回填已运行，但来源没有公开外部 ID 且没有唯一追更候选；
- 已识别；
- 候选冲突。

650 条正式记录全部未识别不自动视为程序错误，但必须能看到最近回填时间及本次扫描、识别、冲突和未变化数量。

## 七、接口兼容

- 现有任务、首页和 RSS 请求保持原路径和基础结构。
- 新增字段均为可选字段，旧客户端忽略后仍能工作。
- `reasonText` 继续存在，但内容改为用户安全文案。
- RSS 旧请求不带目标身份参数时保持原有列表查询行为。
- 目标搜索只增加保守回退，不改变下载信息、详情地址和敏感字段的隐藏规则。

## 验收标准

1. 四条正常保护记录不再出现 `executionState=action_required`，也不进入异常统计。
2. 任务普通界面的任何原因字段均不包含完整路径、Hash、内部 ID 或鉴权 URL。
3. 技术详情仍能查看完整原始原因，方便排障。
4. 首页五条同作品异常分别显示准确季集；`云月大陆 S01E05` 的主因显示为 Symedia 未查询到媒体信息。
5. 身份未关联仅作为次要说明，不遮住真实执行失败。
6. 《清醒点，桃子》等订阅可命中标题可靠、类型不冲突、季号一致或未知的 RSS 候选。
7. 不同 TMDB、明确不同季号或电影年份不一致的种子继续被排除。
8. 订阅资源搜索不再返回大量无关种子，也不会因电视剧标题不含年份而全部返回 0。
9. 有明确 TMDB ID 的已有订阅显示真实海报并写回本地台账；无可靠身份时仍使用占位。
10. 三条固定 RSS 身份样本和一条冲突样本通过完整端到端测试，且正式 RSS 数据库没有测试记录。
11. 正式 RSS 页面能说明最近一次回填是否运行，以及识别、冲突和未变化数量。
12. Python 回归、TypeScript 类型检查、Vite 构建、API 契约和 Compose 解析全部通过。

## 实施边界

实施应保持小步、可独立验证：先修状态和原因，再修首页定位，随后修 RSS 搜索与验收，最后补齐海报。每一步均添加回归测试，不借机调整导航、整体视觉或外部自动化开关。
