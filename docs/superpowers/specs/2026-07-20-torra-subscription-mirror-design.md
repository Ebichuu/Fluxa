# Torra 订阅单向镜像与活动日志闭环设计

日期：2026-07-20

## 1. 背景

Fluxa 当前可以读取 Torra 的订阅总数和运行状态，但“我的订阅”只展示 Fluxa SQLite 中的本地订阅。已有 Torra 订阅不会进入本地台账，因此无法参与 RSS 匹配、任务链追踪和后续自动化。

同时，订阅保存、立即运行等旧兼容写接口受总写入闸门保护；部分失败只返回 HTTP 错误，活动页面又依赖受旧兼容守卫影响的日志接口，导致用户无法在 Fluxa 内追查失败原因。

本阶段建立 Torra 到 Fluxa 的单向订阅镜像，并补齐活动日志闭环。远端删除和双向编辑留到第二阶段。

## 2. 目标

1. 在不修改 Torra 的前提下预览并导入已有订阅。
2. 为每条镜像订阅保存稳定的 Torra 远端 ID，避免重复创建。
3. 让镜像订阅进入 Fluxa 订阅页、RSS 匹配和任务链。
4. 定时更新远端状态，不自动覆盖用户维护的本地展示信息。
5. Fluxa 新订阅经明确确认后推送到 Torra，并保存远端 ID。
6. 为同步、推送、拒绝和失败建立可查询、脱敏的活动记录。

## 3. 非目标

以下能力不属于第一阶段：

- 从 Fluxa 删除 Torra 远端订阅。
- 将 Fluxa 对镜像条目的标题、季数等编辑反向写回 Torra。
- Torra 删除后自动删除 Fluxa 本地记录。
- 启用 115、Telegram、HDHive、MoviePilot 或云盘转存能力。
- 改变 Torra、qBittorrent 或 RSS 的既有写入闸门语义。

## 4. 数据归属

Torra 是已有远端订阅及其运行状态的来源。Fluxa 是本地展示、业务关联和活动记录的来源。

导入的订阅具有以下约束：

- `origin=torra`。
- `readOnly=true`。
- 不会触发 Torra 创建接口。
- 可以参与 RSS 身份匹配和任务链证据关联。
- Torra 暂时不可用时保留最后一次成功同步结果。

Fluxa 新建的订阅仍先写入本地台账。只有管理员明确确认且 `TORRA_PUSH_ENABLED=true` 时，才调用现有 Torra 推送流程。成功后写入远端关联，后续同步与导入订阅使用相同状态模型。

## 5. 数据模型

新增 `torra_subscription_links` 表：

| 字段 | 用途 |
| --- | --- |
| `subscription_key` | Fluxa 稳定订阅 key，主键并关联本地订阅 |
| `remote_id` | Torra 订阅 ID，唯一 |
| `origin` | `torra_import` 或 `fluxa_push` |
| `mapping_status` | `mapped`、`partial` 或 `unmapped` |
| `remote_status` | 脱敏后的远端运行状态 |
| `remote_fingerprint` | 规范化远端身份摘要，用于检测变化 |
| `last_seen_at` | 最近一次在 Torra 中看到该条目的时间 |
| `last_synced_at` | 最近一次成功同步时间 |
| `sync_state` | `current`、`remote_missing` 或 `error` |
| `last_error` | 脱敏错误摘要 |
| `created_at` / `updated_at` | 审计时间 |

现有 `subscriptions.payload_json` 保存展示和业务字段，包括来源、只读状态、媒体身份、远端状态摘要和映射提示。Token、认证头、完整上游响应不得写入数据库。

## 6. 身份映射与去重

同步按以下顺序识别同一订阅：

1. 已保存的 Torra `remote_id`。
2. TMDB ID、媒体类型和季数。
3. 规范化标题、媒体类型、年份和季数。

电影不参与季数比较。剧集在两边都存在有效季数时必须一致。无法可靠识别媒体类型或标题的条目仍可在预览中出现，但标记为 `unmapped`，默认不进入自动化流程。

同一远端 ID 不得关联多个本地订阅。同一本地订阅也不得同时关联多个 Torra ID。冲突写入必须整体回滚并记录活动事件。

## 7. 同步流程

### 7.1 预览

`GET /api/v2/torra/subscription-sync/preview`

该接口只读取 Torra 和本地 SQLite，不写入任何数据。返回：

- Torra 订阅总数。
- 可新增镜像数。
- 已关联数。
- 本地身份重复数。
- 无法映射数。
- 脱敏的冲突摘要。

### 7.2 确认导入

`POST /api/v2/torra/subscription-sync/imports`

请求必须包含：

- `confirm=true`。
- 12 至 128 字符的幂等键。

服务端重新读取 Torra，不能直接信任预览结果。导入在单个 SQLite 事务中完成；同一幂等键重复提交返回原结果，不重复创建订阅。

### 7.3 定时同步

新增 `MCC_TORRA_SUBSCRIPTION_SYNC_ENABLED`，默认关闭，并纳入网页配置。开关由后台循环每次执行前读取，可热生效，不要求重启容器。

开启后由后台调度器每 10 分钟同步一次已关联条目。关闭时后台循环不访问 Torra。同步只更新远端状态、最近出现时间和可验证的进度字段，不创建、编辑或删除 Torra 订阅。

远端条目在一次同步中缺失时标记为 `remote_missing`。本地记录继续保留，后续再次出现时恢复为 `current`。

## 8. 新订阅推送

复用现有 Torra 查重和推送客户端：

1. Fluxa 先保存本地订阅。
2. 用户查看服务端生成的推送预览。
3. 用户明确确认，且 `TORRA_PUSH_ENABLED=true`。
4. 服务端重新查重后创建或触发已有 Torra 订阅。
5. 成功后写入 `torra_subscription_links`。

导入条目因已经具有 `remote_id`，不进入创建流程。任何上游失败都保留本地订阅，并标记为可重试，不回滚用户已保存的本地意图。

## 9. 活动日志

保留现有 JSONL 活动存储，新增不依赖旧兼容总开关的 v2 接口：

- `GET /api/v2/activity/logs`
- `DELETE /api/v2/activity/logs`

第一阶段记录以下事件：

- Torra 同步预览成功或失败。
- 导入开始、成功、部分跳过或失败。
- 定时同步成功、远端缺失或连接失败。
- Torra 推送开始、已有订阅、创建成功、触发搜索失败或被闸门拒绝。
- 订阅和 RSS 写接口返回 403、409、422、502 或 503。

日志元数据只保存本地订阅 key、远端 ID 的短摘要、数量、状态码、稳定错误码和请求 ID。以下内容禁止落日志：密码、Token、Cookie、Authorization、完整 RSS URL、Passkey 和包含查询密钥的外部 URL。

任务中心改用 v2 日志接口，并增加“Torra 同步”筛选。活动项显示用户可理解的中文动作、结果、时间和请求 ID，不直接展示内部异常堆栈。

## 10. 前端行为

订阅页增加同步入口，展示最后同步时间和当前状态：

- 未同步：提供“预览 Torra 订阅”。
- 已预览：展示新增、重复、无法映射数量，并提供确认导入。
- 同步中：禁用重复提交。
- 同步失败：保留上次数据并提供重试。

镜像条目显示“来自 Torra”和只读标识。第一阶段不显示删除或反向编辑动作。第二阶段再增加“移除本地记录”和“删除 Torra 订阅”两个独立动作。

## 11. 错误处理

- Torra 未配置：返回 `TORRA_NOT_CONFIGURED`。
- Torra 认证失败：返回 `TORRA_AUTH_FAILED`。
- 同步开关关闭：写入返回 `TORRA_SUBSCRIPTION_SYNC_DISABLED`。
- 未确认或幂等键非法：返回 400 或 422。
- 身份冲突：返回 409，不产生部分写入。
- Torra 暂时不可用：返回脱敏 502，保留上次同步数据。

所有错误响应包含请求 ID；对应活动事件使用同一请求 ID，便于页面追查。

## 12. 安全边界

- 所有接口继续要求管理员会话。
- 预览不产生外部写入。
- 导入只写 Fluxa SQLite。
- Torra 创建继续由 `TORRA_PUSH_ENABLED` 独立保护。
- 定时同步由 `MCC_TORRA_SUBSCRIPTION_SYNC_ENABLED` 独立保护。
- 第一阶段不实现远端删除方法。
- 不启用 `MCC_PRESERVED_CORE_API_ENABLED` 作为生产解决方案。

## 13. 测试与验收

后端测试覆盖：

- Torra 响应结构规范化。
- 远端 ID、TMDB 和标题降级映射。
- 重复预览和重复导入幂等。
- 冲突事务回滚。
- 远端缺失后本地记录保留。
- 导入条目不会再次推送。
- 开关关闭和认证失败。
- 日志事件完整且敏感字段不泄漏。

前端验证覆盖：

- 预览、确认、加载、成功、部分跳过和失败状态。
- 只读镜像标识及无删除入口。
- 任务中心活动筛选和请求 ID 展示。
- 1920x1080 宽屏布局、浅色和深色主题。

发布前执行完整 Python 测试、TypeScript 检查、生产构建、Compose 解析、契约校验和敏感信息扫描。

## 14. 第二阶段预留

第二阶段单独设计删除能力：

- “移除本地记录”只影响 Fluxa，并保存忽略标记。
- “删除 Torra 订阅”必须使用经过验证的 Torra 删除接口、独立写闸门、二次确认、幂等键和活动审计。
- 两个动作不得共用同一个按钮或默认行为。
