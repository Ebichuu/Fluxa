# 第一阶段订阅工作台设计

日期：2026-07-21

## 目标

将现有订阅页从本地 SQLite 列表升级为可判断、可刷新、可恢复的订阅工作台。页面必须准确表达当前系统能做什么、为什么不能做，以及每条订阅正处于哪一步。

## 范围

- 顶部展示本地写入、Torra 连接、Torra 镜像、RSS、定时任务五项真实能力状态。
- 提供统一刷新、失败重试和最近读取时间。
- 提供电影、剧集、待处理、已入库四项统计。
- 本地台账为空时，引导用户先预览 Torra 订阅，再明确确认导入。
- 每条订阅展示订阅范围、缺集、Torra、qB、115、入库状态和当前阻塞原因。
- 将“执行一轮”改为“更新自动订阅来源”，并明确该动作不等于搜索当前订阅资源。
- 自动订阅来源刷新使用增量合并，不得删除手动订阅、Torra 镜像或不属于本次来源结果的既有条目。

## 第一阶段边界

- 可以读取 Torra 订阅、预览导入、确认写入 Fluxa SQLite，并同步已关联条目的远端状态。
- 不创建远端删除接口，不允许从 Fluxa 删除 Torra 远端订阅。
- Torra 镜像条目保持只读，不显示删除、屏蔽、切季或重复推送操作。
- 页面状态来自实际配置和读取结果；未接入的数据明确显示“未接入”或“无法判断”，不得伪造成功。
- 本轮测试不触发真实下载、远端删除、RSS 写入或 Torra 写入。

## 数据流

新增只读工作台快照接口 `GET /api/v2/subscriptions/workbench`。接口聚合：

1. SQLite 订阅台账及元数据。
2. 运行配置中的写入、RSS、定时任务和镜像开关。
3. Torra 客户端配置与最近一次镜像同步状态。
4. 本地任务链中可匹配到订阅的 Torra、qB、115 和入库证据。

聚合失败采用部分成功：订阅台账仍可返回，单项能力标记为 `error` 并附用户可理解的原因。接口不得因为一个外部服务不可用而隐藏其他可用数据。

## 自动来源合并

自动来源刷新生成本轮候选后，以稳定订阅 key 与现有台账合并：

- 本轮命中的自动条目更新可验证的来源字段。
- 本轮新增自动条目追加到台账。
- 手动条目、Torra 镜像和其他既有条目原样保留。
- 台账元数据记录最近一次自动来源刷新时间和结果统计。

## 前端行为

- 页面首次加载和手动刷新使用同一工作台请求，避免多个请求产生互相矛盾的状态。
- 刷新按钮在按下时立即反馈，保留旧数据直至新数据成功返回。
- 读取失败时保留上次内容，并在原位置提供重试。
- 空台账引导按“预览 → 检查数量和冲突 → 确认导入”顺序呈现。
- 写入关闭时，保存、删除、屏蔽、切季和更新自动来源按钮禁用，并显示关闭原因。
- 动画只使用短促的透明度和位移反馈，并支持 `prefers-reduced-motion`。

## 验收

1. 自动来源刷新不会删除任何既有手动或 Torra 镜像订阅。
2. 工作台一次读取可得到五项能力状态、最后读取时间和四项统计。
3. 空台账可完成 Torra 预览和确认导入引导。
4. 读取失败可原位重试，写能力关闭时操作按钮不可点击。
5. 每条订阅至少显示订阅范围、缺集、Torra、qB、115、入库和阻塞原因；缺少证据时明确显示未知或未接入。
6. Torra 镜像仍不可从 Fluxa 删除或反向编辑。
