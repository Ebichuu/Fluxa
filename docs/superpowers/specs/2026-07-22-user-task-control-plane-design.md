# Fluxa 用户任务中枢设计

日期：2026-07-22  
状态：已批准实施

## 1. 目标

将 Fluxa 从“组件控制台”收敛为面向家庭影音用户的任务中枢。用户不需要理解 Torra、qB、115、Symedia、STRM 和 Emby 的内部职责，也能回答：

- 今天是否正常；
- 哪部作品没有更新以及原因；
- 哪些文件失败了；
- 当前是否可以安全重试。

本设计优先统一信息架构和状态语义，再逐步补齐后端链路证据和安全操作。

## 2. 范围与非目标

### 本次范围

- 重新定义管理页面的职责和导航层级；
- 首页改为面向结果的今日摘要；
- 追更建立 Fluxa 与 Torra 的只读对账模型；
- 任务中心建立统一资源身份和阶段时间线；
- 将异常按用户动作分类，并提供预览后操作；
- 中文化设置，隐藏环境变量名称；
- 保留旧地址跳转，保证收藏和外部链接不失效。

### 明确不做

- 不改影视大厅的视觉结构、播放流程或现有代码边界；
- 第一阶段不从 Fluxa 删除或反向编辑 Torra 远端订阅；
- 不把 Fluxa 做成 Torra、Symedia 或下载器的替代品；
- 不在发现页展示下载器、网盘或入库的技术细节；
- 不一次性重写 Torra 到 Emby 的全部后端链路。

## 3. 页面职责

| 页面 | 核心职责 | 明确不负责 |
| --- | --- | --- |
| 首页 | 给出今天是否正常、入库、下载中、待处理和异常结论 | 不堆完整任务列表，不展示环境变量 |
| 发现 | 找作品、看统一详情、发起追更 | 不展示下载技术细节 |
| 追更 | 维护用户想看的作品，后台对账 Torra | 不再维护第二套“已生效订阅”真相 |
| 任务中心 | 展示每个资源的端到端执行过程和处理入口 | 不放服务配置 |
| 控制室 | 展示服务状态、能力开关、同步和诊断 | 不承担用户偏好和账号表单 |
| 设置 | 管理账号、通知、规则和用户偏好 | 不直接暴露 `MCC_*`、`ENV_*` |
| 影视大厅 | 沉浸式浏览和播放体验 | 不与管理页面混用视觉结构 |

桌面端保留控制室和设置在主导航中。移动端主导航只放“首页、发现、追更、任务中心”，控制室和设置进入右上角管理菜单。影视大厅通过独立入口进入。

旧页面地址保留兼容跳转。跳转时保留查询参数、作品 ID、订阅 ID 和季集定位，避免收藏链接只回到页面首页。

## 4. 订阅事实源与状态维度

### 事实源定义

- Torra 是已经生效的远端订阅事实源；
- Fluxa 保存用户意图、本地镜像、对账结果和操作记录；
- Fluxa 本地独有条目只能表示“仅 Fluxa 存在”或“待推送”，不能直接标记为已生效订阅；
- Torra 独有条目进入“仅 Torra 存在”，允许只读查看和导入镜像。

### 三个独立状态维度

一条订阅同时携带对账、履约和健康状态，三个维度不得互相覆盖：

| 维度 | 状态 |
| --- | --- |
| 对账状态 | 已关联、仅 Fluxa、仅 Torra、存在冲突、远端已消失 |
| 履约状态 | 待同步、追更中、已完成、已暂停、被阻塞 |
| 健康状态 | 正常、等待、正常保护、需要处理、证据不足 |

例如一条订阅可以同时是“已关联 · 追更中 · 正常”，也可以是“仅 Fluxa · 待同步 · 等待用户确认”。

### 第一阶段边界

第一阶段只读获取 Torra 订阅、预览差异、确认导入 Fluxa 镜像并同步远端状态。不得从 Fluxa 删除、屏蔽、切季或反向修改 Torra 远端订阅。对账稳定后，才评估“预览 → 确认”写回能力。

## 5. 统一资源身份与任务链

任务中心不把五个服务的列表简单并排，也不以 `subscription_id` 作为核心身份。手工下载、洗版、补档和 Refind 转存可能没有订阅 ID，统一身份分为三层：

- `media_key`：优先使用媒体类型和 TMDB ID；缺少 TMDB ID 时生成临时身份并标记 `identity_confidence=provisional`；
- `target_key`：媒体与季、集或电影目标；
- `artifact_key`：具体资源，优先使用 qB hash、文件指纹或远端文件 ID。

任务中心另外生成稳定的 `chain_id` 串联各阶段事件。`subscription_id` 只作为来源关联字段。临时身份升级为正式 TMDB 身份时保存别名关系，不静默生成第二条链。

每个资源的阶段证据按以下顺序展示：

`Torra 命中 → qB 下载 → 115 秒传 → Symedia 归档 → STRM 生成 → Emby 索引`

每个阶段都需要记录状态、时间、来源、错误码或保护原因，并包含：

- `observed_at`：实际读到证据的时间；
- `fresh_until`：证据有效期；
- `source`：Torra、qB、115、Symedia、STRM 或 Emby；
- `reason_code`：机器可识别原因；
- `reason_text`：面向用户的中文解释。

缺少日志、数据过期或接口未接入时必须显示“证据不足”，不得归入正常或等待，也不得用接口连通代替阶段完成。

实施时分两段串链：

1. 先接通 `Torra → qB → 115`；
2. 再接入 `Symedia → STRM → Emby`。

## 6. 首页与状态语义

首页首屏只给一个结论，例如：

> 影音中心运行正常  
> 今日入库 14 条 · 下载中 1 条 · 待处理 3 条

存在异常时改为：

> 有 3 个文件上传失败  
> 已重试 2 次，将在 08:00 执行原始上传 · 查看详情

### 五种用户状态

- 正常：所需能力真实可用，最近轮询在预期窗口内，关键资源已完成或正在按计划处理；
- 等待：任务尚未到执行时间、正在下载或等待下一轮重试，不需要用户干预；
- 正常保护：因低分版本拒绝覆盖、已有更高版本或重复资源而跳过，属于策略结果；
- 需要处理：TMDB 无法识别、上传连续失败、依赖掉线、任务超过重试上限或链路出现明确错误。

- 证据不足：缺少关键日志、证据超过 `fresh_until` 或阶段尚未接入，当前无法可靠判断。

单条资源和首页聚合的优先级统一为：

`需要处理 → 证据不足 → 等待 → 正常保护 → 正常`

首页的绿色“正常”只允许由最终媒体结果和调度实际运行状态共同决定。单个服务返回 200、配置已填写或任务线程已创建，都不足以单独判定系统正常。

## 7. 各页面的第一版交互

### 首页

- 今日入库、下载中、待处理、需要处理四项摘要；
- 区分正常保护和真实异常；
- 异常卡片直接进入对应资源时间线；
- 不显示完整任务表。

### 发现

- 榜单、搜索和作品详情；
- 详情页显示追更状态、最近更新和添加追更按钮；
- 资源搜索结果在详情内展开；
- 不出现 qB、115、Symedia 等下游技术字段。

第一阶段点击“添加追更”只创建 Fluxa 意图，成功文案必须是“已保存追更意图，尚未同步到 Torra”。只有远端明确返回成功并保存远端 ID 后，状态才可以变为“已关联”。

### 追更

- 唯一的用户追更列表；
- 显示订阅范围、缺集、对账状态、最近更新和阻塞原因；
- 支持刷新、重试读取和“预览 Torra 差异”；
- 第一阶段不提供远端删除或反向编辑。

### 任务中心

- 以资源为单位展示端到端时间线；
- 先支持 Torra、qB、115 阶段，再补 Symedia、STRM、Emby；
- 每个失败项给出原因分类和下一步建议；
- 不放服务配置表单。

### 控制室

- 继续保留现有四项服务卡片和能力开关；
- 状态文案改成用户可理解的“已连接、未配置、已关闭、最近检查失败”；
- 增加同步、调度、RSS 和诊断证据；
- 管理 Torra、qB、Symedia、Emby 等系统连接、连接测试和能力开关；
- 高级系统配置放在管理员可展开区域；
- 不显示用户偏好或通知模板。

### 设置

- 按“Fluxa 账号、界面偏好、追更偏好、通知对象、通知方式”分组；
- 使用中文名称和影响说明；
- 密码和 Token 已保存时不回显，只有重新输入才修改；
- 不重复提供控制室已经管理的服务地址、Token 或能力开关。

## 8. 异常与安全操作

异常中心把问题分成“需要处理、正常保护、等待状态”。每条异常必须包含：对象、阶段、原因、最近尝试、重试次数、下次计划和可执行动作。

动作不能使用通用“重试当前步骤”实现。每个阶段必须返回动作资格：

- 是否允许；
- 禁止原因；
- 将修改哪些对象；
- 是否支持预览；
- 是否需要确认；
- 幂等键；
- 冷却时间。

可开放的动作限定为：

- 重新检查；
- 重试当前步骤；
- 修改分类；
- 打开原工具；
- 停止追更。

“停止追更”拆成“停止 Fluxa 本地跟踪”和“停止 Torra 远端订阅”。第一阶段只允许前者。

秒传失败但已有计划重试时维持“等待”，不主动提供重试；达到重试上限且原始上传仍失败时才进入“需要处理”。Symedia 低分拒绝属于“正常保护”，禁止重试。Emby 刷新只有在存在新入库证据且冷却结束时才允许。修改分类可能联动 Torra、qB 和秒传路径，不作为第一阶段普通操作。

正常保护仍需展示残留文件，例如“正常保护 · 已保留低分源文件 · 可进入存储清理”。该项不计入红色异常，但不得完全隐藏。

涉及真实写入的动作统一采用：

`预览 → 确认 → 执行 → 留痕`

执行记录保存幂等键、操作者、前后状态、远端响应摘要和可恢复信息。失败不得伪装成成功，也不得在没有确认的情况下批量写入或删除。

## 9. 分阶段实施

### 阶段 A：导航壳和职责收敛

- 调整桌面和移动端导航；
- 重组现有组件；
- 保留旧路由跳转；
- 影视大厅完全不动；
- 先修复调度、RSS、证据时效和状态维度合同。

### 阶段 B：首页结论

- 统一今日摘要接口；
- 修复调度、RSS 和 TMDB 状态误报；
- 增加待处理和异常详情入口。

### 阶段 C：追更只读对账

- 建立 Fluxa/Torra 稳定身份映射；
- 展示 158/112 的差异，而不是硬合并；
- 导入 Torra 镜像前提供预览和冲突说明。

### 阶段 D：任务链第一段

- 统一 `Torra → qB → 115` 资源身份；
- 支持资源时间线和失败重试预览；
- 保留活动记录和恢复证据。

### 阶段 E：任务链第二段与异常中心

- 接入 `Symedia → STRM → Emby`；
- 区分低分保护、重复跳过和真实失败；
- 增加异常分类、重试和刷新媒体库等安全动作。

### 阶段 F：验收与发布

- 以 1920×1080 为主测试深浅主题；
- 覆盖空数据、依赖离线、调度关闭、RSS 采集无命中和部分链路失败；
- 运行前后端测试、类型检查、生产构建和 `git diff --check`；
- 通过 GitHub Actions 先发布版本标签和镜像摘要，冒烟验证通过后再更新 `latest`。

## 10. 验收标准

1. 首页不会在调度关闭时显示“调度已启用、状态正常”。
2. 首页绿色正常代表用户最终媒体链路正常，而不是单个服务连通。
3. 追更页能明确展示 Fluxa 与 Torra 的关联、差异和只读边界。
4. 任务中心能定位到单个资源的失败阶段和下一步动作。
5. 低分拒绝、已有资源跳过和真实失败不会混用同一种状态。
6. 任何真实写操作都经过预览和确认，并保留操作记录。
7. 设置页不再要求普通用户理解环境变量名。
8. 影视大厅视觉和行为不被本次管理工作台改造影响。
9. 缺少证据或数据过期时显示“证据不足”，不得显示绿色正常。
10. Fluxa 本地电影订阅未经确认不得重新写入 Torra。
11. 同一个 Torra 远端订阅重复导入不会生成重复镜像。
12. Torra 远端订阅消失时只标记“远端已消失”，不自动删除本地意图。
13. 正常保护项不提供无意义的重试按钮，但会显示残留文件清理提示。
14. 所有日志脱敏 RSS passkey、Token、Cookie 和外部鉴权参数。
15. 首页按唯一 `target_key` 统计，失败后成功不会重复计算。
16. 工作台和任务列表使用游标分页；日历按月加载，日期详情延迟分页。
17. 每个阶段都能独立上线和回滚，不依赖后续阶段才能使用。
18. 异常中心作为任务中心筛选，不新增主导航入口。
