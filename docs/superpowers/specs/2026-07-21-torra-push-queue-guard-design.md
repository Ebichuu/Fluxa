# Torra 推送排队闸门设计

日期：2026-07-21

## 1. 背景

Fluxa 当前订阅模式为 Torra 时，保存订阅会直接排队后台后处理。即使“允许向 Torra 创建订阅”关闭，后台任务仍会启动，随后被下游安全闸门拒绝，并在活动日志中记录为推送失败。

安全闸门成功阻止了外部写入，但任务中心会被预期内的拒绝记录淹没，用户也容易把“功能未授权”误认为“Torra 故障”。

## 2. 目标

- `TORRA_PUSH_ENABLED=false` 时，保存订阅只写入 Fluxa 本地台账。
- 在排队前停止 Torra 后处理，不创建计时器，不调用 Torra，不记录失败活动。
- 设置页继续使用中文名称“允许向 Torra 创建订阅”；内部配置键不作为主要界面文案。
- 开关开启时保持现有排队、幂等、预览、确认和活动日志行为。

## 3. 实现边界

在 `queue_subscription_resource_rule_transfer()` 解析订阅模式后增加 provider 排队判定。当前变更只处理 `mode=torra` 与 `TORRA_PUSH_ENABLED` 的组合，不推断 MoviePilot、Symedia 或网盘模式的权限语义。

关闭状态返回未排队结果：

- `enabled=false`
- `queued=0`
- `reason="允许向 Torra 创建订阅已关闭"`

调用方继续把订阅保存视为成功，不新增失败活动。Torra 推送预览仍可读取；手动推送仍由现有服务端闸门最终复查。

## 4. 非目标

- 不修改国产平台榜单或筛选。
- 不启用 Torra 订阅镜像，不导入已有 Torra 订阅。
- 不开启自动订阅调度、质量观察、下载、MoviePilot 或云盘能力。
- 不清理历史活动日志。

## 5. 测试

- Torra 模式且开关关闭：不调用 provider、不排队、不写失败活动。
- Torra 模式且开关开启：仍排队一条任务。
- 非 Torra 模式：保持现有行为。
- 运行 TypeScript 检查与全部 Python 回归测试。
