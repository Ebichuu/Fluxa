# Fluxa 产品设计基线

日期：2026-07-23
状态：当前有效

本文只定义 Fluxa 的长期产品职责和数据边界，不规定具体视觉稿。早期 UI 文稿不作为实现依据；页面布局以当前实机、`docs/Fluxa-前端UI改造实施计划.md` 和已确认的专项规格为准。

## 产品定位

Fluxa 是家庭影音任务中枢，不替代 Torra、qBittorrent、115、Symedia 或 Emby。它负责把各系统的事实关联成用户能理解的结论，并在证据充分时提供受控操作。

用户进入 Fluxa 后应能直接回答：

- 今天影音中心是否正常；
- 哪些作品已经更新、正在处理或已经可看；
- 某部作品为什么没有更新；
- 是否需要人工处理，以及允许执行什么操作。

## 页面职责

| 页面 | 核心职责 | 明确不负责 |
| --- | --- | --- |
| 首页 | 给出今天是否正常、成功数量和待办结论 | 不堆完整任务列表 |
| 发现 | 找作品、看详情、保存追更意图 | 不展示下载技术细节 |
| 追更 | 维护追更意图并只读对账 Torra | 不维护第二套远端执行真相 |
| 任务中心 | 展示资源端到端执行过程和异常处理 | 不放服务凭据 |
| 日历 | 展示播出、获取和入库三类时间 | 不用季级证据批量推断单集完成 |
| 控制室 | 服务状态、能力开关、同步和诊断 | 不承担用户偏好 |
| 设置 | Fluxa 账号、连接凭据、规则、通知和偏好 | 不直接暴露环境变量名 |
| 影视大厅 | 沉浸式浏览和播放 | 不与管理工作台混用视觉结构 |

主导航保留首页、发现、追更、任务中心和日历。影视大厅使用独立入口；种子库作为任务中心高级入口；旧路由继续可直接访问。

## 领域事实源

Fluxa 不设一个覆盖全部系统的全局事实源，每个领域只有一个事实源：

- Fluxa：用户追更意图、本地镜像、证据关联和操作记录；
- Torra：真实执行中的订阅和 PT 搜索；
- qBittorrent、115、Symedia：下载、转存和整理过程证据；
- Emby：媒体最终是否可看的事实；
- Fluxa 任务台账：以上证据的关联结果，不改写上游事实。

Torra 单向镜像第一阶段只允许读取、预览、确认导入和同步状态，不允许删除远端订阅。仅 Torra 条目在前后端都保持只读。

## 统一身份

- `mediaKey`：作品身份，优先使用媒体类型和 TMDB ID；
- `targetKey`：电影、季或具体集；
- `chainId`：一次目标执行链；
- `artifactKey`：种子、qB 任务、文件或远端文件；
- `subscriptionId`：来源关联字段，不是任务链核心身份。

同一目标在顶层列表只能出现一次。多个 qB 任务、文件、秒传和入库记录全部挂在同一条链下。一条外部证据最多只能有一个 `ownerTargetKey`；候选冲突时不得绑定。

## 三维状态

状态不得压缩成一个“正常/失败”枚举：

| 维度 | 状态 |
| --- | --- |
| 对账 | 已关联、仅 Fluxa、仅 Torra、存在冲突、远端已消失 |
| 履约 | 待同步、追更中、已完成、已暂停、被阻塞 |
| 健康 | 需要处理、证据不足、等待、正常保护、正常 |

健康优先级为：真实失败优先于证据不足，证据不足优先于等待，等待优先于正常保护，正常保护优先于正常。任何绿色结论都必须有未过期的最终业务证据。

每项状态必须附带 `observedAt`、`freshUntil`、`source`、`reasonCode` 和面向用户的 `reasonText`。缺日志、数据过期或接口未接入时显示“证据不足”，不能显示绿色正常。

任务中心另外保持三个互不覆盖的维度：

- `identityState`：未识别、已关联、候选冲突；
- `executionState`：正常、等待、正常保护、疑似阻塞、需要处理、确认失败；
- `healthState`：决定首页和任务中心的用户介入优先级。

身份未识别不能掩盖已经确认的执行失败；反过来，只有推断出的长时间停滞不能直接冒充红色确认失败，应显示“疑似阻塞”。普通界面使用 `userReasonText`，路径、Hash 和上游原始错误只进入折叠的 `technicalReasonText`。

## 日历状态

日历只用集级证据改变具体剧集状态。过去日期本身不是缺集证据；只有明确属于追更范围、晚于订阅开始或明确纳入补档、超过允许延迟，且没有获取、入库或正常保护证据时，才显示“逾期未获取”。订阅前旧集、范围不明、证据过期或只有季级证据时显示灰色“状态未知”。

## RSS 身份

RSS 身份只接受三类来源：结构化 TMDB/IMDb 字段、简介或公开链接中的明确外部 ID、与现有追更唯一且保守的标准标题匹配。电视剧标题回退必须同时匹配类型和季号；电影必须同时匹配类型和年份。多候选标记冲突，模糊标题不反向认领。

历史身份回填必须由管理员显式触发、限制批量，只处理未识别记录；不得在 GET 查询中隐式写库，不访问 PT 详情页，也不触发下载、转存或推送。

## 安全操作

所有写操作遵循：

`预览 → 确认 → 幂等执行 → 状态复查 → 操作留痕`

每个动作必须返回是否允许、禁止原因、影响对象、是否支持预览、是否需要确认、幂等键和冷却时间。操作资格按阶段计算，不能提供通用“重试当前步骤”。

- 已有计划且未达到重试上限：显示等待，不主动重试；
- Symedia 低分拒绝：显示正常保护，不提供重试；
- Emby 刷新：要求新入库证据和冷却时间；
- 停止本地跟踪与停止 Torra 远端订阅是两个不同动作。

## 展示原则

- 首页只展示用户语言，不显示路径、Hash、内部 Key 或原始响应；
- 技术细节进入任务详情或控制室高级诊断；
- 金色只用于等待或提醒，不同时表示完成；
- 正常保护使用稳定的蓝色语义，真实异常使用红色；
- 管理工作台以清晰、可信和可操作为先；影视大厅保持独立的沉浸视觉；
- 深浅主题、桌面和移动端都必须保持文字对比度和无横向溢出。

## 发布底线

- 缺证据或数据过期不得显示绿色正常；
- 同一证据不得归属于多个目标；
- 没有集级证据的剧集不得显示已获取或已入库；
- 没有明确追更范围和新鲜证据的过去剧集不得显示逾期未获取；
- 首页统计按唯一媒体目标计算，失败后成功不得重复计数；
- 列表使用服务端分页，详情按需加载；
- 日志和响应必须脱敏 Token、Cookie、Passkey 和外部鉴权参数；
- `latest` 只在自动测试、生产构建和冒烟验证通过后更新；
- 每个阶段能够独立上线和回滚。
