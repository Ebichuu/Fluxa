# qBittorrent 可回滚控制动作实施计划

状态：已完成
日期：2026-07-14
设计依据：`docs/superpowers/specs/2026-07-14-qb-reversible-actions-design.md`

## 目标

在任务中心任务卡内增加 qBittorrent 暂停和恢复操作。一次操作当前媒体任务关联的全部 qB 下载，暂停和恢复均需要二次确认；所有操作可审计、可复查，不包含删除或文件变更能力。

## 实施顺序

### 1. 服务端动作基础

- 扩展 `QbittorrentClient`，复用现有登录逻辑调用 qB 批量暂停/恢复接口。
- 新增 `QbittorrentActionService`，负责 hash 格式、数量、存在性、目标状态和批量动作校验。
- 动作提交后重新读取 qB 状态，区分成功、跳过、失败和“已提交但未确认”。
- 复用 `ActivityLog` 记录开始、成功、跳过和失败，只保留脱敏 hash。

验证：模拟 qB 服务断言登录、写接口、状态复查和无真实外部调用。

### 2. API 契约

- 在现有 `/api/qbittorrent` 下新增：
  - `POST /api/qbittorrent/actions/pause`
  - `POST /api/qbittorrent/actions/resume`
- 成功返回 `200`；动作已提交但无法确认返回 `202`。
- 输入错误返回 `400`，任务不存在返回 `404`，qB 未配置或离线返回 `503`，上游动作失败返回 `502`。
- 错误响应保持项目现有 `{ error }` 兼容形状，并增加机器可读 `code`。

验证：路由不把错误包装成 `200`，不暴露上游 Cookie、密码或响应正文。

### 3. 任务链控制状态

- 在 `TaskChainItem` 增加 qB 控制摘要：关联数量、暂停数量、是否可暂停、是否可恢复。
- 本地订阅主干行和未关联 qB 行使用同一计算逻辑。
- 不改变四步任务链、任务状态计数和现有关联规则。

验证：增加混合状态、全部暂停和无 qB 关联测试。

### 4. 任务中心交互

- 仅有 qB 关联的任务卡显示操作按钮。
- 存在未暂停任务时显示“暂停下载”；全部暂停时显示“恢复下载”。
- 增加工作页风格确认层，显示媒体标题、关联下载数量和动作影响。
- 执行期间按钮禁用；完成后刷新统一任务链；错误不做乐观更新。
- 手机端操作区保持单栏，四步证据链继续使用 2×2。

验证：构建通过，三档视口无横向溢出，键盘焦点和执行状态可辨认。

### 5. 文档与安全校验

- 更新 `README.md`、`docs/PLAN.md` 和 `docs/IMPLEMENTATION_SOURCES.md`。
- 保持 `TORRA_PUSH_ENABLED=false`。
- 检查真实账号、密码、Cookie、Token 和完整 qB hash 未写入仓库。
- 确认影院大厅、媒体队列、顶部导航和 Mineradio iframe 未修改。

## 真实环境边界

本阶段不对局域网 qBittorrent 执行暂停或恢复。接口只通过模拟服务验证；真实动作必须由用户后续明确授权并指定测试任务。
