# 家庭媒体控制中心运行说明

更新时间：2026-07-13

## 总体架构

家庭媒体控制中心是一条“自动发现资源 → 筛选版本 → 下载做种 → 上传 115 → 自动整理 → 生成 STRM → Emby 播放”的流水线。

```text
PT 站 / RSS / RSS Cloud
        ↓
Torra：订阅匹配、版本控制、资源评分
        ↓
qBittorrent：下载并持续做种
        ↓
file_event_mover：建立硬链接
        ↓
Torra secupload_115 + p115client：秒传/原始上传
        ↓
115 /00-待整理/<分类>
        ↓
CloudDrive2：报告文件创建事件
        ↓
Symedia：识别、命名、洗版、归档
        ↓
115 /媒体库
        ↓
Webhook / AutoSymlink：创建 STRM
        ↓
Emby：展示和播放
```

Torra 管上游资源决策，Symedia 管下游媒体整理，Hermes 负责巡检、提醒和长期维护。

## 一、Torra：资源决策中心

Torra 从 PT 站、RSS 和 RSS Cloud 获取候选资源，然后依次处理：

1. 识别影片、剧集、季集和发布组。
2. 统一分辨率、视频编码、音频编码、动态范围、来源、体积等资源事实。
3. 匹配订阅目标。
4. 先按版本控制规则分桶。
5. 再按资源权重评分和排序。
6. 排除媒体库已有集数。
7. 把最终选中的资源推送给 qBittorrent。

Torra 决定“下载什么、选择哪个版本、种子中的哪些文件需要下载”。

## 二、qBittorrent：下载和做种

qB 下载到：

```text
/qbdownload/torra/<分类>
```

当前共 8 个分类：`00-日漫`、`01-国漫`、`02-国产剧`、`03-日韩剧`、`04-欧美剧`、`05-港台剧`、`06-综艺`、`10-电影`。

下载完成后，`file_event_mover` 不移动原文件，而是在下面建立硬链接：

```text
/qbdownload/秒传/<分类>
```

因此 qB 原文件仍留在 `/torra/<分类>` 持续做种；秒传流程可以独立移动或删除自己的目录项。

## 三、Torra 秒传：送入 115

Torra 的 `secupload_115` 负责监听、缓存、失败状态、重试、回退和本地清理，真正与 115 通信的是 `p115client`。

正常顺序：

1. 查询进程内 SHA1 缓存。
2. 查询持久化 SHA1 缓存。
3. 两级缓存均未命中时读取文件计算 SHA1。
4. 调用 `P115Client.upload_file_init()` 尝试秒传。
5. 成功后将文件创建到 115：

```text
/00-待整理/<分类>
```

6. 115 确认成功后，按 `delete_by_success=true` 删除秒传侧硬链接；qB 原文件不受影响。

秒传失败时：

1. 使用 `rename()` 把秒传侧目录项移入 `/qbdownload/秒传失败/<分类>`。
2. 建立待处理记录，首次失败为 `attempts=1`。
3. `retry_pending` 定时继续秒传。
4. 当前配置达到 3 次失败后，在同一轮切换到 `P115Client.upload_file()` 原始上传。
5. 原始上传成功后清理本地入口和待处理记录；仍失败则保留文件与状态，等待下一轮。

当前 `retry_pending` 每 8 小时执行一次。

## 四、CloudDrive2：115 与 Symedia 的事件桥梁

文件进入 115 的 `/00-待整理` 后，CloudDrive2 产生文件创建事件。Symedia 的 `CloudDriveMessageListener` 接收事件，再交给 `ArchiveWatcher`。

因此主归档链路由文件事件即时触发，不依赖定时扫描。

## 五、Symedia：识别、洗版和归档中心

Symedia 收到文件后负责：

1. TMDB 和媒体信息识别。
2. 判断电影、剧集、季和集。
3. 生成规范化目录和文件名。
4. 检查历史记录和运行时重复任务。
5. 执行版本控制与权重洗版。
6. 通过 CloudDrive2 移动到正式媒体库。
7. 写入成功或失败历史。

归档路径：

```text
/CloudNAS/CloudDrive/115/00-待整理/<分类>
        ↓ cd2_move
/CloudNAS/CloudDrive/115/媒体库
```

### 权重洗版规则

目标目录有多个同版本候选时，Symedia 先选择分数最低的目标，然后执行严格比较：

```text
源文件分数 > 最低目标分数  → 覆盖
源文件分数 <= 最低目标分数 → 不覆盖
```

同分不会覆盖，也没有按体积、时间或文件名继续裁决的分支。

特殊规则：

- “总是覆盖”只给源文件增加 `99999 × 权重`，目标文件该项权重强制为 0。
- 规则分数为 `-1` 时，新源文件转换为 `-99999`，用于阻止不合格资源洗入；目标旧文件仍按 `-1` 计算。
- `-1` 排除项的权重必须保持正数，负权重会把极低哨兵反向变成巨额正分。

## 六、STRM 和 Emby

Symedia 完成 `cd2_move` 后，CloudDrive2 产生重命名事件。Webhook/AutoSymlink 根据事件在以下目录创建对应 STRM：

```text
/CloudNAS/STRM/115/媒体库
```

Emby 读取 STRM 媒体库，实际视频继续保存在 115。这样本地不需要保存完整媒体库，同时保留规范目录、元数据、海报和正常播放体验。

## 七、清理与维护

各清理动作相互独立：

- 秒传成功：只删除秒传侧硬链接。
- qB 原文件：继续保留并做种。
- 达到清理条件后：由 `torrent_remover` 和 Hermes 安全清理任务处理。
- Symedia：保留归档历史、洗版详情和失败记录。
- Hermes：检查秒传队列、失败目录、重试任务、归档状态、STRM 结果和存储空间。

## 组件职责总结

| 组件 | 职责 |
|---|---|
| Torra | 发现资源、订阅匹配、版本控制、权重评分和下载决策 |
| qBittorrent | 下载、文件选择和持续做种 |
| file_event_mover | 从 qB 分类目录向秒传目录建立硬链接 |
| p115client | 执行 115 秒传和原始上传 |
| CloudDrive2 | 映射 115、执行云盘文件操作并传递文件事件 |
| Symedia | 媒体识别、命名、洗版、归档和历史记录 |
| Webhook / AutoSymlink | 生成 STRM 并联动媒体库更新 |
| Emby | 最终媒体展示与播放入口 |
| Refind | 根据缺失内容和规则补充资源，再进入 Symedia 链路 |
| Hermes | 巡检、提醒、自动清理和长期维护 |
