# 管理员认证

## 目标

Fluxa 使用首次启动创建管理员的认证方式，不再要求用户配置访问密钥、允许来源或 Cookie 安全开关。

- 第一次打开 Fluxa 时创建唯一管理员账号和密码。
- 初始化完成后关闭注册入口，只保留管理员登录。
- 账号、密码摘要和会话密钥保存在持久目录，升级或重启不会丢失。
- HTTP 内网和 HTTPS 反向代理自动选择正确的 Cookie 属性。
- 忘记密码时通过容器内交互命令重置，不把新密码写进命令历史。

当前版本只支持一个管理员，不增加普通用户、邀请注册、邮箱找回、OAuth 或第三方登录。

首次初始化前任何能访问服务的人都可能抢先创建管理员。部署后应先在受信局域网完成初始化，再配置公网反向代理；这是取消引导密钥后的明确剩余风险。

## 首次初始化

认证状态独立保存在 `db/auth.sqlite3`，不与订阅台账混用。数据库只包含一个管理员记录和一份随机会话密钥。

未创建管理员时：

1. 所有页面请求跳转到 `/auth/setup`。
2. 初始化页面要求填写管理员账号、密码和确认密码。
3. 账号长度为 3–32 个字符，密码至少 8 个字符。
4. 密码使用 Werkzeug `scrypt` 摘要保存，不保存明文或可逆密文。
5. SQLite 唯一约束和事务保证并发初始化只能成功一次。
6. 初始化成功后直接建立管理员会话并进入首页。

管理员已经存在时，`GET /auth/setup` 跳转到登录页，`POST /auth/setup` 返回 `409 ADMIN_ALREADY_INITIALIZED`。

## 登录与会话

登录页使用管理员账号和密码。连续五次失败后按来源地址锁定 15 分钟，保留当前限速语义。

会话 Cookie：

- 名称继续使用 `mcc_session`。
- `HttpOnly`、`SameSite=Strict`、有效期七天。
- 当前请求为 HTTPS 时自动增加 `Secure`；HTTP 内网访问不增加。
- 会话由持久化随机密钥签名，修改或重置密码时轮换密钥，使旧会话立即失效。

`GET /api/auth/session` 返回：

- `setupRequired`：是否尚未创建管理员。
- `authenticated`：当前会话是否有效。
- `username`：已登录管理员账号，否则为 `null`。
- `expiresAt`：会话到期时间，否则为 `null`。

## 请求来源处理

删除 `MCC_ALLOWED_ORIGINS` 手工白名单，不再要求用户填写 IP、域名或端口。

管理员认证不再按 `Origin`、Host 或 `Sec-Fetch-Site` 拒绝请求，因此直接访问、反向代理和地址变化都不需要额外配置。服务端不会向任意来源返回带凭据的 CORS 响应；受保护 API 仍必须持有有效管理员会话，浏览器会话 Cookie 使用 `HttpOnly` 与 `SameSite=Strict`。

## 密码重置

管理员忘记密码时在部署目录执行：

```bash
docker compose exec fluxa python -m app.admin reset-password
```

命令在终端中交互读取管理员账号、新密码和确认密码，不回显密码。成功后更新密码摘要、轮换会话密钥并注销所有现有会话。

不提供网页免认证重置入口，也不通过环境变量传递新密码。

## 升级迁移

从旧访问密钥版本升级时：

- 业务数据和外部服务配置保持不变。
- `MCC_ACCESS_KEY`、`MCC_ALLOWED_ORIGINS`、`MCC_COOKIE_SECURE` 停止作为认证配置使用。
- 如果 `db/auth.sqlite3` 不存在，首次访问进入管理员初始化页面。
- 旧 `mcc_session` Cookie 自动失效，不迁移旧访问密钥为管理员密码。

该变更属于认证方式调整，发布版本使用 `v0.3.0`。

## 验收标准

- 空持久目录首次访问只能进入管理员初始化页面。
- 初始化完成后无法再次注册管理员。
- 数据库和日志不出现管理员明文密码。
- 重启容器后管理员账号和登录能力保持有效。
- HTTP 内网登录不需要 `MCC_COOKIE_SECURE=false`。
- HTTPS 反向代理登录自动使用 Secure Cookie。
- 不配置任何 Origin 环境变量也能完成登录和受保护操作。
- 反向代理或访问地址变化不会返回 `ORIGIN_FORBIDDEN`。
- 密码重置命令使全部旧会话失效。
- 现有 API、React 页面、影院大厅和 172 项回归保持兼容。
