# Fluxa 前端 UI 改造实施计划

日期：2026-07-23
状态：阶段 H 已发布；阶段 I 事实源一致性收口已完成（2026-07-24）
范围：Fluxa 管理工作台、任务数据契约、前端 UI、交互与响应式体验

本计划整合当前实机检查、用户任务中枢产品方案和任务链/API 审查结论。早期 UI 设计稿及 `docs/UI_STANDARD.md` 不再作为实施依据；后续设计只参考当前产品职责、真实数据状态和本计划。

- 当前自动化验收已通过：304 项 Python 回归、v2 53 条机器契约、TypeScript 类型检查和 Vite 生产构建通过。此前桌面、390×844 移动端、深浅主题、月/周日历和 44px 触控区已验收；本轮已完成首页季集定位、普通/技术原因分层、正常保护和资源搜索反馈的本地只读冒烟。本机没有可用于复现的正式 RSS/本地订阅数据，追更补图与 RSS 三类身份识别由隔离自动化样本覆盖；真实家庭影音链路冒烟仍在发布后进行。

## 1. 改造目标

将 Fluxa 从偏工程化的组件控制台，调整为面向家庭影音用户的任务中枢。

用户进入 Fluxa 后，应当能够快速回答：

- 今天影音中心是否正常；
- 今天有哪些作品更新、正在获取或已经入库；
- 哪部作品没有更新以及原因；
- 当前任务进行到下载、115、Symedia、STRM 或 Emby 的哪个阶段；
- 哪些情况属于正常等待或保护，哪些情况需要人工处理；
- 当前操作是否可以安全执行。

本次改造优先统一导航、页面职责、状态语义和交互方式，再进行视觉精修。

## 2. 范围与边界

### 2.1 本次实施范围

- 重构管理工作台导航；
- 建立统一设计变量与公共 UI 组件；
- 重构首页、发现、追更、日历、任务中心、控制室和设置；
- 建立统一的正常、等待、保护、异常和证据不足视觉语义；
- 完成桌面端、平板端和移动端适配；
- 增加加载、空数据、依赖离线、证据过期和部分失败状态；
- 完善弹窗、抽屉、确认、结果反馈和键盘操作；
- 保留旧页面地址和定位参数。

### 2.2 明确不做

- 不重做影视大厅现有视觉和播放流程；
- 不改变 Torra、qBittorrent、115、Symedia、STRM 和 Emby 的业务职责；
- 不改变外部组件职责，也不重写下载或入库引擎；但必须先修正 Fluxa 自身的任务聚合、分页、摘要和详情契约，否则前端不得自行伪造唯一任务链；
- 不把 Fluxa 做成另一套下载器、网盘工具或 Symedia；
- 不继续使用旧版 `docs/UI_STANDARD.md` 作为设计依据；
- 不在普通用户页面默认展示环境变量、Hash、内部 Key 和原始接口响应。

## 3. 产品与页面职责

| 页面 | 核心职责 | 不负责 |
| --- | --- | --- |
| 首页 | 回答今天是否正常，汇总入库、更新、等待和异常 | 不堆完整任务列表，不展示工程字段 |
| 发现 | 查找作品、浏览榜单、查看详情、保存追更意图 | 不展示下载器和网盘技术状态 |
| 追更 | 管理想看的作品，查看缺集、履约和 Torra 对账 | 不展示完整执行日志 |
| 日历 | 展示什么时候播出、何时获取、何时入库 | 不承担订阅规则编辑和任务诊断 |
| 任务中心 | 展示单个资源的端到端执行阶段和处理入口 | 不放服务连接配置 |
| 控制室 | 查看服务连接、能力、同步、调度和诊断证据 | 不放普通用户偏好 |
| 设置 | 管理服务连接凭据、账号、规则、通知、界面和用户级配置 | 不承担实时故障诊断 |
| 影视大厅 | 保持现有沉浸式浏览和播放体验 | 不与管理工作台混用视觉结构 |

## 5. 统一视觉体系

### 5.1 设计变量

统一定义：

- 页面背景；
- 一级、二级和悬浮表面；
- 主文字、次要文字和弱化文字；
- 边框、分隔线和焦点边框；
- 品牌强调色；
- 五种状态颜色；
- 圆角、阴影、间距、动画时长；
- 桌面、平板和移动端断点。

### 5.2 状态颜色

| 用户状态 | 颜色 | 含义 |
| --- | --- | --- |
| 正常 | 绿色 | 已完成或系统真实可用 |
| 等待 | 黄色 | 正在下载、等待执行或未到重试时间 |
| 正常保护 | 蓝紫色 | 低分拒绝、已有更高版本或重复跳过 |
| 需要处理 | 红色 | 明确失败、超过重试上限或依赖不可用 |
| 证据不足 | 灰色 | 无法确认状态或证据已经过期 |

同一种状态在所有页面使用相同颜色、图标和文案。状态不能只依靠颜色区分。

### 5.3 字体与密度

- 正文不小于 14px；
- 辅助文字不小于 12px；
- 宽屏页面标题控制在 40～48px，移动端控制在 30～34px；
- 面板标题控制在 18～24px，卡片标题控制在 14～17px；
- 桌面按钮可点击高度不低于 40px；
- 移动端点击区域不低于 44px；
- 工程字段使用等宽字体，但默认折叠；
- 减少无实际信息的英文眉题和装饰文案。

### 5.4 卡片与层级

- 页面背景不使用强阴影；
- 摘要卡片使用轻边框；
- 主要结果卡片使用轻阴影；
- 弹窗和菜单使用较强层级；
- 半透明和模糊效果只用于导航、弹窗和少量关键卡片；
- 不让所有卡片同时使用玻璃效果和重阴影。

## 6. 公共组件实施

### 6.0 后端数据契约前置

前端改造开始前，先完成以下只属于 Fluxa 聚合层的收口，不修改 Torra、qBittorrent、115、Symedia、STRM 或 Emby 的业务职责：

1. 同一 `targetKey` 在顶层只返回一个稳定 `chainId`；多个订阅、qB 任务、文件和入库记录作为该链路的来源或 artifact 合并。
2. 健康状态按“需要处理 → 证据不足 → 等待 → 正常保护 → 正常”聚合，不以最新一条记录覆盖更严重证据。
3. 新增轻量摘要 `GET /api/v2/tasks/summary`。
4. `GET /api/v2/tasks/chains` 支持 `healthState`、`chainId`、`targetKey`、`updatedAfter`、`offset` 和 `limit`，默认返回 20 条摘要。
5. 新增 `GET /api/v2/tasks/chains/{chainId}`，展开任务时再读取阶段证据、artifact 和动作资格。
6. 列表响应包含 `total`、`offset`、`limit`、`hasMore` 和稳定 `version`；支持 `ETag / If-None-Match`，无变化返回 `304`。
7. 使用服务端共享快照，避免每个浏览器轮询都重新探测全部外部服务并写入全量账本。
8. 旧 `/api/tasks/chain` 保留兼容；v2 前后端在同一版本迁移。

首页、任务中心和顶部状态只能消费上述聚合结果，不在浏览器内重新统计完整任务数组。

### 6.1 页面骨架

整理或建立：

- `AppShell`；
- `AppTopNav`；
- `MobileNavigation`；
- `ManagementMenu`；
- `PageHeader`；
- `PageContainer`。

要求：

- 桌面以 1920×1080 为主要设计基线，使用全宽工作台和稳定页边距，不收窄为居中的传统后台容器；
- 导航高度固定；
- 当前页面选中状态清楚；
- 管理菜单点击外部或按 ESC 关闭；
- 页面切换保留必要的筛选和定位参数。

### 6.2 状态组件

优先建立：

- `StatusBadge`；
- `StatusDot`；
- `HealthBanner`；
- `MetricCard`；
- `EmptyState`；
- `LoadingState`；
- `EvidenceStaleNotice`；
- `ErrorState`。

页面只传入标准状态，不允许各页面自行决定状态颜色。

### 6.3 交互组件

建立：

- `ConfirmDialog`；
- `ActionPreviewDialog`；
- `Toast`；
- `AdvancedDetails`。

`Drawer`、`SearchField`、`FilterTabs` 和 `DatePicker` 只在三个以上页面出现真实重复后抽取，不为建立组件库而提前重写页面。

所有弹窗与抽屉支持：

- ESC 关闭；
- 键盘操作；
- 焦点锁定与焦点恢复；
- 清楚的取消和确认按钮；
- 执行中禁止重复提交；
- 成功与失败分别反馈。

## 7. 首页改造

### 7.1 页面结构

1. 今日运行结论；
2. 今日更新提醒；
3. 四项核心摘要；
4. 需要处理事项；
5. 最近入库和处理结果；

### 7.2 四项核心摘要

- 今日入库；
- 正在获取；
- 等待处理；
- 需要处理。

首页不直接使用“qBittorrent 下载中”或“115 秒传队列”等组件名称。具体服务放到任务详情。

### 7.3 今日结论

允许显示：

- 影音中心运行正常；
- 部分任务正在等待；
- 部分能力受限；
- 有任务需要处理；
- 当前证据不足，无法判断。

- 没有真实异常；
- 调度状态可信；
- 关键证据没有过期；
- 最终媒体结果正常。

## 10. 日历页改造

日历是 Fluxa 核心页面，独立保留。

### 10.1 页面目标

清楚展示：

- 作品预计什么时候播出；
- Fluxa 什么时候找到资源；
- 什么时候完成本地入库；
- 哪些剧集已经更新但还没有获取成功。

### 10.2 三类时间

- 播出时间：剧集预计或实际发布的时间；
- 获取时间：Fluxa/Torra 找到并开始处理资源的时间；
- 入库时间：完成 Symedia、STRM 和 Emby 入库的时间。

界面不得把三种时间统一写成模糊的“已更新”。

### 10.3 日历状态

| 状态 | 显示方式 |
| --- | --- |
| 即将播出 | 灰色日期标记 |
| 今日更新 | 蓝色高亮 |
| 正在获取 | 黄色状态点 |
| 已入库 | 绿色勾选 |
| 正常保护 | 蓝色盾牌 |
| 超时未获取 | 红色提醒 |
| 时间未知 | 灰色问号 |

### 10.4 页面结构

顶部提供：

- 月视图和周视图切换；
- 上一周期、今天、下一周期；
- 全部、即将更新、正在获取、已入库和缺集筛选；
- 作品搜索；
- “仅看我的追更”开关。

日期格展示：

- 作品名称；
- 季集；
- 状态图标；
- 多个作品时显示前几项和剩余数量。

点击日期后打开当日详情抽屉，点击剧集后展示：

- 作品与季集；
- 预计播出时间；
- 当前追更状态；
- 是否已经找到资源；
- 是否完成入库；
- 最近证据时间；
- 查看追更详情；
- 查看任务链。

### 10.5 时间规则

- 所有时间统一使用 `Asia/Shanghai`；
- 明确区分日期未知、时间待定和证据过期；
- 同一剧集的日历状态与追更、任务中心保持一致。

## 11. 任务中心改造

涉及页面：`TasksCenter.tsx`

这是前端 UI 改造的最高优先级页面。

### 11.1 资源聚合

后端保证每个 `targetKey` 只返回一条顶层链路，前端每条链路展示一张资源卡片：

- 作品名称和季集；
- 当前阶段；
- 用户状态；
- 最近更新时间；
- 原因说明；
- 下一步建议；
- 查看详情入口。

同一资源的多个文件和版本由详情接口返回，放入“相关文件”折叠区。前端不得通过数组去重掩盖后端重复链路。

### 11.2 阶段时间线

桌面端使用横向阶段条：

`资源命中 → 下载 → 115 → Symedia → STRM → Emby`

移动端使用纵向时间线。

每个阶段展示：

- 用户可理解的名称；
- 完成、等待、失败或未知；
- 最近证据时间；
- 原因说明；

#### 任务状态拆分

任务顶层不能把身份问题和执行结果压成同一个健康状态。保留三个字段组：

- `identityState`：`unidentified`、`linked`、`conflict`；
- `executionState`：`normal`、`waiting`、`protected`、`suspected_blocked`、`action_required`、`confirmed_failed`；
- `healthState`：继续用于首页颜色和用户介入优先级。

阶段存在 `action_required` 但仅因为身份缺失、长时间没有后续证据时，顶层不得直接计入红色“需要处理”，应计入独立的“疑似阻塞”摘要，并在任务详情显示：已完成阶段、等待时长、缺失阶段、推断依据和下一次可观察时间。明确上传失败或入库失败仍进入“需要处理”；身份冲突仍显示“证据不足/冲突”，不能伪装成正常。

首页和任务中心摘要至少返回 `suspectedBlocked`，任务筛选提供“疑似阻塞”；原有五态字段继续返回，旧客户端不因新增字段失败。

#### 日历未知与逾期分离

月摘要不得仅凭“播出日期已过且没有证据”生成“逾期未获取”。只有同时满足以下条件才允许标记 `missing`：

1. 该集明确属于用户追更范围；
2. 已经过播出时间和配置的允许延迟；
3. 没有获取、入库或正常保护证据。

订阅建立前已播出的旧集、订阅范围不明、证据未刷新或仅有季级证据的条目改为灰色 `unknown`（状态未知）。当日详情继续显示具体原因，月摘要按 `unknown` 统计；旧日历请求不带新参数时保持原结构，新增状态字段可选。

#### RSS 身份验收

- RSS 结构化字段直接包含 TMDB ID，结果为 `identified`；
- RSS 简介或公开链接包含 IMDb ID，结果为 `identified`；
- 没有外部 ID，但与已有追更唯一可靠匹配，结果为 `identified` 且记录来源与置信度。

三条样本全部通过后，648 条历史未识别记录才可以进入受控回填；多候选仍保持 `conflict`，模糊标题不反向认领追更身份。

## 19. 秒传证据与统计可信度收口（2026-07-23）

### 19.1 已确认的 Torra 只读契约

Torra 当前通过 `GET /api/v1/plugins/secupload_115` 提供插件视图，响应包含：

- 秒传配置项及启用状态；
- `scan_source` 与 `retry_pending` 两类任务定义；
- 每个分类的定时计划、最近运行和下一次执行时间；
- 最近运行状态、开始/结束时间及“成功 N 个、失败 N 个”的任务级摘要。

该接口当前没有提供逐文件队列、文件失败次数、原始上传状态或单文件下一次重试时间。Fluxa 第一阶段只读接入已有字段，不根据配置目录、qB 完成时间或 Symedia 缺失反推出不存在的逐文件状态。

### 19.2 状态规则

- 有 Symedia 源文件记录，只能证明文件已经进入下游处理链，不能反推具体秒传方式；
- 有 Torra 插件活动运行，可显示“秒传插件正在处理”，但不能绑定到某个作品，除非响应提供唯一文件身份；
- 最近运行成功或失败属于分类级运行证据，单独展示，不直接覆盖单个任务健康状态；
- qB 已完成但没有 Torra 秒传或下游证据时，`cloud115` 阶段为 `unknown/missing`；
- 只有 Torra 返回的明确失败、达到重试上限或原始上传失败才能进入红色需要处理；
- 正常保护继续保持蓝色，不进入失败统计。

### 19.3 首页与任务中心

- 首页不再展示由时间差推断出的“疑似阻塞”数量；
- 大量未关联记录合并成一个系统级证据不足提示：`任务身份尚未完成关联，当前无法准确判断秒传积压`；
- 任务中心顶部展示 Torra 秒传插件是否可读、活动运行数量、最近重试结果和下一次计划；

### 19.4 后续串链

资源身份继续按以下顺序收口：Torra 订阅 ID → qB Hash → 分类相对路径 → 文件名与季集 → Symedia 源路径 → Emby 季集证据。当前已实现 Torra 已归属文件名与 Symedia `src/dest/files` 的完全一致匹配；仅以归一化完整文件名建立唯一归属，不采用包含关系或标题回退。qB 当前没有与 Torra 可验证的 Hash、路径或文件名交集，继续保持未关联。每条证据先收集候选，再统一裁决；没有唯一所有者时保持未关联或冲突，不抢占其他目标。

### 19.5 Torra 只读追更与日历

- 当 Fluxa 本地订阅为空时，任务链直接读取 Torra 远端订阅作为只读目标；同 TMDB、同季的本地条目优先，不产生重复目标。
- Torra 订阅完成仅表示远端追更生命周期完成，不等同于 Symedia 或 Emby 已入库；工作台分别显示“Torra 订阅已完成”和实际入库证据。
- 日历将 `only_torra` 且具有完整 TMDB 身份的条目纳入播出时间线。远端创建时间缺失时，历史集一律显示状态未知，不推断为缺集。
- Torra 只读订阅、任务目标和日历均不写入 Fluxa 本地台账，不启用反向推送、删除或远端修改。
