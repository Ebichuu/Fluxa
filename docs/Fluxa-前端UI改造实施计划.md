# Fluxa 前端 UI 改造实施计划

日期：2026-07-22
状态：已完成本地实施，待发布后实机冒烟
范围：Fluxa 管理工作台、任务数据契约、前端 UI、交互与响应式体验

本计划整合当前实机检查、用户任务中枢产品方案和任务链/API 审查结论。早期 UI 设计稿及 `docs/UI_STANDARD.md` 不再作为实施依据；后续设计只参考当前产品职责、真实数据状态和本计划。

## 当前实施进度

截至 2026-07-22：

- 阶段 0、A、B 已完成：任务按 `chainId/targetKey` 唯一聚合，共享快照、摘要/分页/详情、`updatedAfter`、ETag、完整阶段进度、技术详情和五态公共组件均已落地。
- 安全操作已收口：qB 暂停/恢复和证据驱动的 Emby 刷新完成预览、确认、复查、幂等和留痕；Torra 秒传失败重试与 Symedia 失败文件重新归档没有真实上游写接口，页面明确显示当前不可在 Fluxa 内执行，未伪造操作按钮。
- 阶段 C、D 已完成：首页使用轻量业务结论并精准跳转任务；追更分离对账、履约和健康三维状态，支持 Torra 只读差异预览、确认导入、远端消失保留本地意图和动态追更文案。
- 阶段 E 已完成：日历通过 `/api/v2/calendar` 聚合播出、获取和入库证据，提供月/周视图、当日详情、状态/作品筛选与追更/任务精准跳转；导航保留宽屏日历入口和移动端四项主入口。
- 阶段 F 已完成：控制室分开配置、连通和业务健康，读取真实调度/推送状态并提供脱敏高级诊断；设置是服务地址、凭据和持久规则的唯一编辑入口，技术字段默认隐藏，敏感值不回显。
- 阶段 G 的本地验收已完成：238 项 Python 回归、v2 51 条机器契约、TypeScript 类型检查、Vite 生产构建、变更完整性、质量和安全扫描通过。1920×1080、1366×768 和 390×844 的已有深浅主题基线保持；发布后真实家庭影音链路冒烟仍需在新镜像部署后执行。
- 全程未执行真实 Torra 删除、下载、秒传、归档、批量推送或其他外部写操作。

## 1. 改造目标

将 Fluxa 从偏工程化的组件控制台，调整为面向家庭影音用户的任务中枢。

用户进入 Fluxa 后，应当能够快速回答：

- 今天影音中心是否正常；
- 今天有哪些作品更新、正在获取或已经入库；
- 哪部作品没有更新以及原因；
- 当前任务进行到下载、115、Symedia、STRM 或 Emby 的哪个阶段；
- 哪些情况属于正常等待或保护，哪些情况需要人工处理；
- 当前操作是否可以安全执行。

本次改造优先统一导航、页面职责、状态语义和交互方式，再进行视觉精修。

## 2. 范围与边界

### 2.1 本次实施范围

- 重构管理工作台导航；
- 建立统一设计变量与公共 UI 组件；
- 重构首页、发现、追更、日历、任务中心、控制室和设置；
- 建立统一的正常、等待、保护、异常和证据不足视觉语义；
- 完成桌面端、平板端和移动端适配；
- 增加加载、空数据、依赖离线、证据过期和部分失败状态；
- 完善弹窗、抽屉、确认、结果反馈和键盘操作；
- 保留旧页面地址和定位参数。

### 2.2 明确不做

- 不重做影视大厅现有视觉和播放流程；
- 不改变 Torra、qBittorrent、115、Symedia、STRM 和 Emby 的业务职责；
- 不改变外部组件职责，也不重写下载或入库引擎；但必须先修正 Fluxa 自身的任务聚合、分页、摘要和详情契约，否则前端不得自行伪造唯一任务链；
- 不把 Fluxa 做成另一套下载器、网盘工具或 Symedia；
- 不继续使用旧版 `docs/UI_STANDARD.md` 作为设计依据；
- 不在普通用户页面默认展示环境变量、Hash、内部 Key 和原始接口响应。

## 3. 产品与页面职责

| 页面 | 核心职责 | 不负责 |
| --- | --- | --- |
| 首页 | 回答今天是否正常，汇总入库、更新、等待和异常 | 不堆完整任务列表，不展示工程字段 |
| 发现 | 查找作品、浏览榜单、查看详情、保存追更意图 | 不展示下载器和网盘技术状态 |
| 追更 | 管理想看的作品，查看缺集、履约和 Torra 对账 | 不展示完整执行日志 |
| 日历 | 展示什么时候播出、何时获取、何时入库 | 不承担订阅规则编辑和任务诊断 |
| 任务中心 | 展示单个资源的端到端执行阶段和处理入口 | 不放服务连接配置 |
| 控制室 | 查看服务连接、能力、同步、调度和诊断证据 | 不放普通用户偏好 |
| 设置 | 管理服务连接凭据、账号、规则、通知、界面和用户级配置 | 不承担实时故障诊断 |
| 影视大厅 | 保持现有沉浸式浏览和播放体验 | 不与管理工作台混用视觉结构 |

## 5. 统一视觉体系

### 5.1 设计变量

统一定义：

- 页面背景；
- 一级、二级和悬浮表面；
- 主文字、次要文字和弱化文字；
- 边框、分隔线和焦点边框；
- 品牌强调色；
- 五种状态颜色；
- 圆角、阴影、间距、动画时长；
- 桌面、平板和移动端断点。

### 5.2 状态颜色

| 用户状态 | 颜色 | 含义 |
| --- | --- | --- |
| 正常 | 绿色 | 已完成或系统真实可用 |
| 等待 | 黄色 | 正在下载、等待执行或未到重试时间 |
| 正常保护 | 蓝紫色 | 低分拒绝、已有更高版本或重复跳过 |
| 需要处理 | 红色 | 明确失败、超过重试上限或依赖不可用 |
| 证据不足 | 灰色 | 无法确认状态或证据已经过期 |

同一种状态在所有页面使用相同颜色、图标和文案。状态不能只依靠颜色区分。

### 5.3 字体与密度

- 正文不小于 14px；
- 辅助文字不小于 12px；
- 宽屏页面标题控制在 40～48px，移动端控制在 30～34px；
- 面板标题控制在 18～24px，卡片标题控制在 14～17px；
- 桌面按钮可点击高度不低于 40px；
- 移动端点击区域不低于 44px；
- 工程字段使用等宽字体，但默认折叠；
- 减少无实际信息的英文眉题和装饰文案。

### 5.4 卡片与层级

- 页面背景不使用强阴影；
- 摘要卡片使用轻边框；
- 主要结果卡片使用轻阴影；
- 弹窗和菜单使用较强层级；
- 半透明和模糊效果只用于导航、弹窗和少量关键卡片；
- 不让所有卡片同时使用玻璃效果和重阴影。

## 6. 公共组件实施

### 6.0 后端数据契约前置

前端改造开始前，先完成以下只属于 Fluxa 聚合层的收口，不修改 Torra、qBittorrent、115、Symedia、STRM 或 Emby 的业务职责：

1. 同一 `targetKey` 在顶层只返回一个稳定 `chainId`；多个订阅、qB 任务、文件和入库记录作为该链路的来源或 artifact 合并。
2. 健康状态按“需要处理 → 证据不足 → 等待 → 正常保护 → 正常”聚合，不以最新一条记录覆盖更严重证据。
3. 新增轻量摘要 `GET /api/v2/tasks/summary`。
4. `GET /api/v2/tasks/chains` 支持 `healthState`、`chainId`、`targetKey`、`updatedAfter`、`offset` 和 `limit`，默认返回 20 条摘要。
5. 新增 `GET /api/v2/tasks/chains/{chainId}`，展开任务时再读取阶段证据、artifact 和动作资格。
6. 列表响应包含 `total`、`offset`、`limit`、`hasMore` 和稳定 `version`；支持 `ETag / If-None-Match`，无变化返回 `304`。
7. 使用服务端共享快照，避免每个浏览器轮询都重新探测全部外部服务并写入全量账本。
8. 旧 `/api/tasks/chain` 保留兼容；v2 前后端在同一版本迁移。

首页、任务中心和顶部状态只能消费上述聚合结果，不在浏览器内重新统计完整任务数组。

### 6.1 页面骨架

整理或建立：

- `AppShell`；
- `AppTopNav`；
- `MobileNavigation`；
- `ManagementMenu`；
- `PageHeader`；
- `PageContainer`。

要求：

- 桌面以 1920×1080 为主要设计基线，使用全宽工作台和稳定页边距，不收窄为居中的传统后台容器；
- 导航高度固定；
- 当前页面选中状态清楚；
- 管理菜单点击外部或按 ESC 关闭；
- 页面切换保留必要的筛选和定位参数。

### 6.2 状态组件

优先建立：

- `StatusBadge`；
- `StatusDot`；
- `HealthBanner`；
- `MetricCard`；
- `EmptyState`；
- `LoadingState`；
- `EvidenceStaleNotice`；
- `ErrorState`。

页面只传入标准状态，不允许各页面自行决定状态颜色。

### 6.3 交互组件

建立：

- `ConfirmDialog`；
- `ActionPreviewDialog`；
- `Toast`；
- `AdvancedDetails`。

`Drawer`、`SearchField`、`FilterTabs` 和 `DatePicker` 只在三个以上页面出现真实重复后抽取，不为建立组件库而提前重写页面。

所有弹窗与抽屉支持：

- ESC 关闭；
- 键盘操作；
- 焦点锁定与焦点恢复；
- 清楚的取消和确认按钮；
- 执行中禁止重复提交；
- 成功与失败分别反馈。

## 7. 首页改造

涉及页面：`Overview.tsx`

### 7.1 页面结构

1. 今日运行结论；
2. 今日更新提醒；
3. 四项核心摘要；
4. 需要处理事项；
5. 最近入库和处理结果；
6. 查看日历、追更和任务中心入口。

### 7.2 四项核心摘要

- 今日入库；
- 正在获取；
- 等待处理；
- 需要处理。

首页不直接使用“qBittorrent 下载中”或“115 秒传队列”等组件名称。具体服务放到任务详情。

### 7.3 今日结论

允许显示：

- 影音中心运行正常；
- 部分任务正在等待；
- 部分能力受限；
- 有任务需要处理；
- 当前证据不足，无法判断。

绿色正常必须同时满足：

- 没有真实异常；
- 调度状态可信；
- 关键证据没有过期；
- 最终媒体结果正常。

接口加载失败或证据不足时，不能显示绿色正常。

### 7.4 页面联动

- 点击“今日更新”进入当天日历；
- 点击异常进入对应资源任务；
- 点击最近入库进入作品或任务详情；
- 首页数字必须与任务中心筛选结果一致。

## 10. 日历页改造

日历是 Fluxa 核心页面，独立保留。

### 10.1 页面目标

清楚展示：

- 作品预计什么时候播出；
- Fluxa 什么时候找到资源；
- 什么时候完成本地入库；
- 哪些剧集已经更新但还没有获取成功。

### 10.2 三类时间

- 播出时间：剧集预计或实际发布的时间；
- 获取时间：Fluxa/Torra 找到并开始处理资源的时间；
- 入库时间：完成 Symedia、STRM 和 Emby 入库的时间。

界面不得把三种时间统一写成模糊的“已更新”。

### 10.3 日历状态

| 状态 | 显示方式 |
| --- | --- |
| 即将播出 | 灰色日期标记 |
| 今日更新 | 蓝色高亮 |
| 正在获取 | 黄色状态点 |
| 已入库 | 绿色勾选 |
| 正常保护 | 蓝色盾牌 |
| 超时未获取 | 红色提醒 |
| 时间未知 | 灰色问号 |

### 10.4 页面结构

顶部提供：

- 月视图和周视图切换；
- 上一周期、今天、下一周期；
- 全部、即将更新、正在获取、已入库和缺集筛选；
- 作品搜索；
- “仅看我的追更”开关。

日期格展示：

- 作品名称；
- 季集；
- 状态图标；
- 多个作品时显示前几项和剩余数量。

点击日期后打开当日详情抽屉，点击剧集后展示：

- 作品与季集；
- 预计播出时间；
- 当前追更状态；
- 是否已经找到资源；
- 是否完成入库；
- 最近证据时间；
- 查看追更详情；
- 查看任务链。

### 10.5 时间规则

- 所有时间统一使用 `Asia/Shanghai`；
- 明确区分日期未知、时间待定和证据过期；
- 同一剧集的日历状态与追更、任务中心保持一致。

## 11. 任务中心改造

涉及页面：`TasksCenter.tsx`

这是前端 UI 改造的最高优先级页面。

### 11.1 资源聚合

后端保证每个 `targetKey` 只返回一条顶层链路，前端每条链路展示一张资源卡片：

- 作品名称和季集；
- 当前阶段；
- 用户状态；
- 最近更新时间；
- 原因说明；
- 下一步建议；
- 查看详情入口。

同一资源的多个文件和版本由详情接口返回，放入“相关文件”折叠区。前端不得通过数组去重掩盖后端重复链路。

### 11.2 阶段时间线

桌面端使用横向阶段条：

`资源命中 → 下载 → 115 → Symedia → STRM → Emby`

移动端使用纵向时间线。

每个阶段展示：

- 用户可理解的名称；
- 完成、等待、失败或未知；
- 最近证据时间；
- 原因说明；
- 可执行动作。

未获得真实证据时显示“未知”或“未接入”，不能假定完成。

### 11.3 工程字段

以下内容默认隐藏到“技术详情”抽屉：

- `mediaKey`；
- `targetKey`；
- `chainId`；
- `artifactKey`；
- Hash；
- 原始错误码；
- 幂等键；
- 原始响应摘要。

技术详情提供复制按钮，但不占据默认任务卡片。

### 11.4 筛选

- 全部；
- 需要处理；
- 等待；
- 正常保护；
- 已完成；
- 证据不足；
- 阶段筛选；
- 时间范围；
- 作品搜索。

### 11.5 操作安全

重新检查、重试、修改分类、清理文件等操作统一使用：

`操作说明 → 影响预览 → 确认 → 执行状态 → 结果留痕`

清理文件不得点击后直接执行。界面必须展示操作对象、影响范围和不可恢复风险。

## 12. 控制室改造

涉及页面：`ControlRoom.tsx`

### 12.1 服务状态

每个服务展示：

- 已连接；
- 未配置；
- 已关闭；
- 最近检查失败；
- 证据已过期。

服务连接成功不等于完整业务链路正常，不能仅凭单个服务返回成功显示“主链正常”。

### 12.2 默认信息

- 服务状态；
- 最近检查时间；
- 当前影响；
- 重新检查；
- 打开原工具。

### 12.3 高级诊断

默认折叠：

- 原始日志；
- 接口地址；
- 响应代码；
- 调度证据；
- RSS 证据；
- 同步详情。

日志中的 Token、Cookie、Passkey 和密码必须脱敏。

## 13. 设置页改造

涉及页面：`SettingsPage.tsx`

按以下分组：

- 服务连接；
- 账号与安全；
- 界面偏好；
- 追更规则；
- 资源获取；
- 通知渠道；
- 高级选项。

要求：

- 不直接显示环境变量名；
- 每个开关说明开启后的实际影响；
- 保存后显示明确结果；
- Token 和密码不回显；
- 配置失败不能显示保存成功；
- 控制室负责观察、诊断和临时操作；
- 设置负责持久化 Torra、qBittorrent、Symedia、Emby 等服务地址与凭据，以及账号、规则、通知和偏好；
- 同一个服务地址和凭据只能在设置中编辑一次，控制室只显示脱敏后的连接状态与测试结果。

## 18. 验收标准

1. 用户进入首页后五秒内能够判断影音中心是否正常；
2. 首页、日历、追更、任务中心和控制室不存在状态矛盾；
3. 首页异常可以直接进入对应资源；
4. 用户能从日历看清播出、获取和入库时间；
5. 日历可以清楚识别今日更新、正在获取、已入库和缺集；
6. 任务列表不会因多个文件版本重复展示同一集；
7. 普通页面默认不展示工程字段；
8. 正常保护不会被显示为失败；
9. 证据不足不会被显示为正常；
10. 危险操作都有预览、确认和结果反馈；
11. 仅 Torra 存在的条目保持只读；
12. 移动端能够完成查看日历、追更、定位异常和安全重试；
13. 深浅主题和所有主要状态清晰可读；
14. 影视大厅现有视觉、播放流程和代码边界不受影响；
15. 旧页面地址、日期定位、作品定位和任务深链接仍然有效；
16. 正式版本不存在模拟状态按钮和虚假成功提示。
17. 同一个 `chainId/targetKey` 在任务列表中只出现一次，重复来源全部进入详情。
18. 任务列表默认最多返回 20 条摘要，展开详情时才读取阶段证据和 artifact。
19. 相同任务版本的条件请求返回 `304`，轮询不得重复传输完整响应。
20. 任务列表响应包含稳定分页字段，首页统计直接来自服务端摘要。
21. 1920×1080 为主要视觉验收视口，同时通过 1366×768、768px、430px 和 390px。
22. 每个视口只保留导航和一个主要浮层使用磨砂效果，嵌套卡片不叠加模糊。

## 19. 最终实施原则

实施顺序保持为：

> 先统一任务身份和服务端事实，再完成任务中心与首页，然后改造追更、日历、导航和视觉。

Fluxa 的前端不以展示更多工程数据为目标，而是让用户更快理解家庭影音系统当前发生了什么、下一步会发生什么，以及何时需要自己处理。
