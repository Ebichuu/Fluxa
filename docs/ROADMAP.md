## 2. 当前下一阶段：PT 主链实机闭环

主线固定：

```text
媒体控制中心订阅
  → Torra
  → qBittorrent
  → Torra 秒传 115
  → Symedia
  → Emby
```

实施顺序：

1. fnOS 单容器只读部署。
2. 核对 Torra 下载器 ID、八分类保存路径和 115 秒传目标。
3. 创建一条电影测试订阅，只读取 Torra 推送预览。
4. 用户单独批准后开启一次 Torra 推送。
5. 观察 qB 下载、Torra 秒传、Symedia 接管和 Emby 索引证据。
6. 电影闭环成立后验证一条剧集季。
7. 最后开启订阅调度。

当前不执行以上实机动作。

## 3. MoviePilot 其他 PT 站点补齐

MoviePilot 不是 Telegram 网盘通道。后续目标：

- Torra 永远优先。
- 只有 Torra 明确无结果或达到等待条件，且没有 Torra/qB 活动任务时，才允许 MoviePilot 参与。
- MoviePilot 下载继续进入同一 qB → 115 → Symedia → Emby 主线。
- 不允许 Torra 与 MoviePilot 对同一媒体并行下载。
- 使用现有 `moviepilot_status()`、`moviepilot_subscribe()`，并以 [MoviePilot 官方文档](https://wiki.movie-pilot.org/) 核对接口，不猜测。
- 当前仅保留配置和状态，自动补齐状态机尚未设计。

## 4. Telegram 网盘能力（延期但保留）

未来范围仍包括：

- Telegram 登录、频道选择和频道资源订阅。
- HDHive / pansou 授权、搜索、签到和影巢解锁。
- 115 分享链接候选预览与单条转存。
- PT 失败后的人工或自动网盘兜底。
- 真实 115 转存直接证据、失败对账和防重复。
- 115 清理和助力；这两项风险较高，继续排在最后。

保留边界：

- Python 模块、原路由、v2 安全适配层和模拟测试不得删除。
- 当前 React 不引用这些动作接口。
- 所有环境闸门保持关闭。
- 不启动 NasEmby `legacy/ptto115.py`；PT 主线秒传由 Torra 独占。
- 未来重新启用前必须另写设计并重新核对 Torra、qB、115、Symedia 和 Emby 重复证据。

详细历史设计与未完成步骤见 `docs/CLOUD_ACQUISITION_PLAN.md`。

## 5. 其他未完成工作

### 任务链证据

- 获取 Torra 秒传到 115 的直接结果；接口没有该证据时继续明确标记推断。
- 无法可靠关联的外部记录继续显示未关联，不猜测合并。
- 实机确认八分类在 qB 保存路径、115 待整理目录和 Symedia 归档之间一致。

### 受控动作

- qB 暂停/恢复真实任务验证。
- Emby 证据刷新真实验证。
- Torra 单条真实推送与冷却验证。

### 部署

- fnOS 单容器只读部署。
- HTTPS 反向代理和源站端口限制。
- 持久目录备份、升级和回滚演练。
- 多 worker 或多副本前增加调度选主和台账并发控制。

## 6. 明确不做

- 不建立网页播放器；播放交给 Emby 客户端。
- 不建立 PT 站点爬虫；PT 搜索交给 Torra，未来补齐交给 MoviePilot。
- 不让 NasEmby `ptto115.py` 与 Torra 秒传同时运行。
- 不向 Symedia 重复推送订阅；Symedia 只负责 115 后整理入库。
- 不把 NasEmby 原静态管理页面作为第二套用户界面启用。
- 不建立第二套订阅台账或第二套调度器。
