## 2. 下一项代码主线：网盘获取通道

用户确认网盘订阅必须保留。最终策略是：

- PT / Torra 为默认主通道。
- 网盘通道由全局和订阅级开关控制，默认关闭。
- PT 没有可用资源、达到等待阈值或用户手动选择时，才进入网盘候选流程。
- 网盘结果转存前必须复查订阅、Torra、qB、115、Symedia 和 Emby 证据，避免重复获取。
- 当前保留的 115、Telegram、HDHive / pansou、Symedia 和 NasEmby provider 模块是这条通道的业务基础，不删除。
- 原接口用途、调用函数、副作用和接入状态见 `docs/CORE_API_CAPABILITY_MATRIX.md`。

详细计划见 `docs/CLOUD_ACQUISITION_PLAN.md`。

## 3. 其他未完成代码工作

### 服务连接管理

- 在媒体控制中心系统设置中展示 115、Telegram、HDHive / pansou 和 Symedia 的配置状态。
- 浏览器只显示掩码与连接状态，不接收服务端凭据。
- 登录、频道、账号等敏感配置使用单独安全接口和强确认，不复用原 NasEmby 管理页面。

### 任务链补全

- 显示 PT 搜索、网盘搜索、候选确认、转存、Symedia 处理和 Emby 索引证据。
- 标记“PT 等待”“允许网盘兜底”“网盘处理中”“已转存”“等待入库”和“重复已阻止”。
- 无法可靠关联的记录继续显示未关联，不猜测合并。

### 受控动作

- qB 暂停/恢复真实任务验证。
- Emby 证据刷新真实验证。
- 网盘搜索、转存和重试动作的确认、幂等、冷却和审计。

### 部署

- fnOS 单容器只读部署。
- HTTPS 反向代理和源站端口限制。
- 持久目录备份、升级和回滚演练。

## 4. 明确不做

- 不建立网页播放器；播放交给 Emby 客户端。
- 不建立 PT 站点爬虫；PT 搜索交给 Torra。
- 不把 NasEmby 原静态管理页面作为第二套用户界面启用；源码和调用关系继续保留。
- 不建立第二套订阅台账或第二套调度器。
- 不让 PT 与网盘通道同时无条件获取同一媒体。
